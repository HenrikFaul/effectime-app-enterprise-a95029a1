## One-sentence user-language description

When an authorized manager creates or approves leave on someone's behalf, the
same operation can be retried safely after a lost connection or app restart
without creating the same request twice.

## Problem it replaces

A successful server commit could previously be followed by a lost response,
leaving the manager unsure whether the action happened. Retrying had no durable
end-to-end identity and could duplicate the leave request, quota effect,
approval or audit trail. The new workflow preserves a privacy-minimized retry
receipt and reconciles the exact same operation instead of guessing.

## Primary beneficiary persona(s)

- **HR manager** — completes delegated leave administration with a clear,
  retryable outcome.
- **Workspace owner / resource assistant** — avoids duplicate requests when a
  connection drops during a privileged action.
- **Payroll and compliance operator** — receives one consistent quota and audit
  effect for the same identified operation.
- **IT / security administrator** — gets a fail-closed, traceable workflow that
  does not persist the raw justification in the retry outbox.

## Notes / caveats for marketing

- Source merge and hosted CI are proven; production rollout is not.
- Exactly-once applies only to calls carrying the same v2 actor/workspace/key
  and canonical payload. It is not a global promise across different devices,
  different keys or legacy v1 clients.
- The seven-day value is a retry/reconciliation horizon, not an automatic data
  deletion TTL.
- Runtimes without the required Web Locks boundary fail closed for this
  privileged operation; Lockdown Mode is a documented limitation.
- Do not use "zero duplicates in every situation", "offline leave approval" or
  "production available" until the missing rollout evidence exists.
- This capability does not by itself certify GDPR, SOC 2 or ISO 27001
  compliance.

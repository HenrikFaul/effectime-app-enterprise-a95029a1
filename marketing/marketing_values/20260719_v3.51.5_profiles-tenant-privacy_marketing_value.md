## One-sentence user-language description

The web and mobile candidate can show authorized teams upcoming birthdays and
work anniversaries from one canonical workspace source without exposing user
IDs, birth years, profile preferences or another organization's people.

## Problem it replaces

The old dashboard assembled milestones by downloading more profile data than it
needed and filtering it in the browser. That was not a tenant privacy boundary,
made errors look like empty results, and allowed date edge cases around midnight,
daylight-saving changes and leap years. The candidate moves authorization and
data minimization to one server contract and gives the shared client explicit
loading, error and retry behavior.

## Primary beneficiary persona(s)

- **HR manager** — keeps useful employee milestones in the team workspace
  without distributing full profile preferences.
- **Team lead / operations manager** — sees the same current milestone view on
  desktop and mobile.
- **Employee** — benefits from a smaller, purpose-specific disclosure of their
  profile data.
- **Data protection / security lead** — gains a testable tenant and column
  boundary instead of browser-side filtering.
- **IT administrator** — operates one shared Supabase contract rather than a
  separate mobile data store or privileged mobile API.

## Claim types enabled

- **Privacy / data minimization** — returns only workspace membership ID,
  display name, milestone type, month and day required by the UI; no user ID.
- **Compliance support** — enforces workspace and field-level access through a
  testable server contract.
- **Visibility** — provides a consistent upcoming milestone list across web and
  native shells.
- **Coordination** — helps authorized teams acknowledge birthdays and work
  anniversaries using current workspace membership data.
- **Error reduction** — explicit retry/error states and exact calendar handling
  replace false-empty and date-coercion behavior.
- **Capacity integrity** — ambiguous duplicate names and email-only assignees stay
  visibly unmatched instead of attributing or double-counting work against the
  wrong person.
- **Platform consistency** — web, Android and iOS use the same React adapter and
  Supabase source of truth.
- **Global identity integrity** — a user may update only their own global display
  name; one workspace administrator cannot silently rename them in every tenant.

## Notes / caveats for marketing

- The feature is a candidate and is not deployed to production.
- Milestone access requires active workspace membership, both calendar and
  members read permission, and the `birthday_widget` plus `members_list`
  entitlements. It is not a public directory.
- The separate same-workspace coworker directory is intentionally available for
  organizational name/avatar resolution without members-administration
  permission. Its browser projection is only `user_id`, display name and avatar;
  it exposes no locale, email, milestone or preference data.
- The milestone RPC returns exactly membership ID, display name, milestone type,
  month and day for active targets. It does not expose user ID, birth year,
  email, raw preferences, activation state or temporary-access material.
- Own locale and global display-name updates are self-only RPC operations. A
  future administrator-managed alias must be workspace-owned metadata rather
  than a write to another user's global profile.
- Android and iOS use the same candidate React/Supabase source, but this is not
  a claim that signed store applications have been released.
- Production remains blocked by the 59/68/84 migration-history/schema drift,
  restored-staging acceptance and missing live Git-SHA deployment attestation.
- The privacy boundary intentionally breaks the old raw-preferences widget after
  DB apply. Production therefore needs a controlled maintenance rollout with
  immediate client release; do not market this as seamless backward compatibility.
- Do not claim "GDPR certified", "all profile data encrypted", "available in
  app stores" or "zero privacy risk". This boundary supports data minimization;
  it does not independently establish regulatory certification.

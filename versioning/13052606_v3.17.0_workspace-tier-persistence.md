# v3.17.0 — Workspace tier persistence + visibility + superadmin-only change

**Date:** 2026-05-13
**Branch:** `claude/review-project-requirements-lR0Wp`

## Reported concern (from the user)

> "Létrehozok egy workspacet egy kiválasztott tier-ben, és random
> változik utána mikor újra megnyitom. ez súúúúúúúúúúúlyos bug. fizetős
> felhasználók pillanat alatt fognak kártérítési pert indítani."

Customer-paid-tier accidentally downgrading would indeed be catastrophic.
Investigation findings:

1. **No code path writes `tenant_subscriptions.tier_id` after creation.**
   The `create_workspace_with_owner` RPC inserts the chosen tier at
   creation time and that row is never updated by anything in `src/`,
   `supabase/functions/`, any trigger, any RLS policy, any cron job, or
   any migration (other than the one-shot v3.13 backfill on 2026-05-12
   which only fills in workspaces missing a subscription row).
2. **But the active tier was invisible everywhere in the product UI.**
   Users had no way to verify their tier post-creation. They could only
   infer it from which features were available — and feature visibility
   depends on the tier PLUS addons PLUS overrides PLUS a 5-minute client
   cache. The combination felt random.
3. **There was no admin-side UI to change a tier post-creation** —
   not even for platform admins.

So this release ships persistence enforcement + visibility + admin control,
all three.

## DB primitives (applied via Supabase MCP migration `workspace_tier_visibility_and_admin_change`)

### `public.workspace_active_tier` (view)

Maps `workspace_id` → current active subscription row joined to the tier:

```sql
CREATE VIEW public.workspace_active_tier
WITH (security_invoker = on) AS
SELECT tw.workspace_id, tw.tenant_id, ts.id AS subscription_id,
       t.id AS tier_id, t.tier_key, t.name AS tier_name,
       t.sort_order AS tier_sort_order, ts.seats, ts.started_at, ts.ends_at
FROM public.tenant_workspaces tw
JOIN public.tenant_subscriptions ts ON ts.tenant_id = tw.tenant_id
                                    AND ts.status = 'active'
                                    AND (ts.ends_at IS NULL OR ts.ends_at > now())
JOIN public.tiers t ON t.id = ts.tier_id;
```

`security_invoker` so RLS on the underlying tables applies — members see
only their own workspace's tier, platform admins see all.

### `public.superadmin_change_workspace_tier(uuid, text, text)`

The ONLY supported way to mutate a workspace's tier after creation.

- **AuthN:** `auth.uid()` must be non-null.
- **AuthZ:** `has_role(auth.uid(), 'admin'::app_role)` must return true.
  Enterprise owner / resourceAssistant is NOT enough — tier changes have
  billing impact and must flow through the platform team.
- **Action:** UPDATE the active `tenant_subscriptions` row (or INSERT one
  if the workspace somehow has none); stamp `metadata` with
  `last_tier_change_at/actor/reason/from/to`.
- **Audit:** writes `platform_audit_events` row with action
  `workspace_tier_changed`, `target_type='workspace'`, prev/new state.
- **Returns:** `jsonb{ ok, workspace_id, tenant_id, subscription_id,
  from_tier_key, to_tier_key }`.

```sql
REVOKE ALL ON FUNCTION ... FROM PUBLIC;
GRANT EXECUTE ON FUNCTION ... TO authenticated;
```

The RPC re-checks the admin role server-side, so revoking PUBLIC is
defense in depth.

## Frontend

### `src/hooks/useWorkspaceTier.ts` (new)

- `useWorkspaceTier(workspaceId)` — React Query hook returning the active
  tier for a single workspace. 60s stale time.
- `fetchWorkspaceTiers(ids[])` — batch fetch returning a Map for the
  picker grid (one round-trip).

### `WorkspaceTierBadge` (new, in `WorkspaceDashboard.tsx`)

Small visible badge next to the workspace name in the dashboard header.
Color-coded by tier: freemium = muted gray, pro = blue, enterprise+ =
amber. Tooltip says "Subscription tier: X. Only a platform admin can
change this." Reads via `useWorkspaceTier`.

### Picker-card tier badge (in `Enterprise.tsx`)

Each card in the workspace picker now shows the tier alongside the role
badge. Batch-fetched once when the picker loads.

### Superadmin → Workspaces tab

- **New "Tier" column** with a color-coded badge per row.
- **New "Change tier…" dropdown menu item.** Opens a dialog with:
  - Current tier display
  - New-tier dropdown (loaded via the new `list-tiers` action)
  - Reason input (free-text, stored in audit log)
  - Submit calls `change-workspace-tier` action which delegates to the
    superadmin_change_workspace_tier RPC.
  - Submit is disabled if the user picks the current tier (pre-empts
    useless audit entries).
- After successful change, the workspace list reloads, the new tier shows
  in the Tier column, and the user sees a toast like "Tier changed: pro
  → enterprise".

### Edge function `superadmin-hub` → v3

Adds 2 actions:
- `change-workspace-tier` — delegates to the RPC. Verifies platform admin
  role twice (once in the edge function, once in the RPC).
- `list-tiers` — used by the change-tier dialog.

All 10 prior actions preserved.

## Governance update

`.governance/ui_ux_rules.md`: new "Core principle: Workspace tier
persistence" section. CLAUDE.md quick-reference updated. These now define:
- Tier is set ONCE at creation; only `superadmin_change_workspace_tier`
  may change it.
- No edge function, hook, RPC, trigger, RLS policy, or migration may
  modify `tenant_subscriptions.tier_id` outside of those two writers.
- The tier badge in the dashboard header is mandatory.
- Demo workspaces follow the same rule — the operator's selection in
  `CreateWorkspaceDialog` persists; the `seed-demo-workspace` default
  `'enterprise'` only applies when no `tier_key` is sent.

## Why no data migration was needed

All 12 existing workspaces are on `freemium`. None were silently
downgraded by code. The user's perception of "random tier changes" was
caused by tier-invisibility + tier-determined feature visibility, not by
actual mutation. This release fixes the visibility gap and locks down
the write path so the perception can be verified true going forward.

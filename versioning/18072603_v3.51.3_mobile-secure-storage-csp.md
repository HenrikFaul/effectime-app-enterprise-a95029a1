# v3.51.3 — Mobile secure storage and CSP hardening (unreleased)

Date: 2026-07-18
Status: local implementation; no store, frontend, Edge or database release

## Requirement

**BIZONYÍTOTT:** the Android and iOS Effectime foundations must use the same
Supabase data source as the web product without storing native authentication
sessions in an unprotected WebView store. A native release also requires a
WebView policy that does not inherit the broader public-web SEO/PWA surface.

## Implemented

- Exact-pinned `@aparajita/capacitor-secure-storage@8.0.0` adapter. Web keeps its
  existing storage contract; native runtime has no localStorage or memory
  fallback.
- Stable explicit Supabase project/storage identity and a strict allowlist for
  the session, PKCE verifier and optional user keys.
- Project-bound schema-v1 secure envelope, serialized read-modify-write, exact
  write/readback verification and cache update only after commit.
- Crash-safe legacy migration: secure commit precedes verified legacy cleanup;
  secure state wins conflicts. An iOS sandbox marker prevents a Keychain session
  surviving uninstall from silently restoring a previous installation.
- Supabase-compatible auth mutex, including the structural `isAcquireTimeout`
  contract used by non-blocking auto-refresh ticks.
- Blocking recovery UX with safe diagnostic code and two-step local-session
  deletion. Plugin, OS and token details are not logged.
- Terminal reset latch rejects late sign-in writes; secure and legacy cleanup
  targets are attempted independently. A verified empty secure tombstone and
  guarded delete fallback prevent marker/legacy cleanup failure from reviving a
  stale session after restart.
- Logout checks the Supabase result and atomically purges local credentials when
  remote revocation fails, with an explicit degraded-state warning.
- Native custom-scheme and HTTPS links reject raw session tokens on every route;
  legacy implicit restoration remains web-only.
- Existing web M365 redirect behavior remains available, while native Connect is
  fail-closed disabled at both UI and handler level with a localized message in
  all eight supported locales. Native system-browser callback parity remains open.
- Separate `dist-mobile` build with exact HTTPS/WSS Supabase CSP. It contains no
  wildcard Supabase source, `unsafe-eval`, inline script, JSON-LD, service worker
  or web-only PWA assets. Public web SEO/PWA output remains separate.
- Portable post-sync Swift path normalization, Android/iOS full-tree SHA-256
  artifact equality and exact plugin allowlist checks. The transitive
  Capacitor Keyboard package is present only in the lock graph and is not a
  registered native plugin.

No database schema, RLS, RPC, Edge or public API contract changed.

## Supply-chain and platform risk

The secure-storage package is MIT and compatible with Capacitor 8, but has a
single-maintainer concentration risk. Exact version/integrity, lockfile, SBOM,
npm audit and native plugin allowlist are release requirements. Browser E2E does
not prove AndroidKeyStore/Keychain behavior or OS WebView/WKWebView CSP injection.

## Compatibility and rollback

- Web login/callback/storage compatibility and SEO/PWA behavior are unchanged;
  shared logout error handling intentionally became fail-closed.
- Android and iOS use the same backend/data contracts and identical mobile
  artifact.
- Source/config/package changes are revertible as one reviewed unit before store
  release. Secure-session reset remains explicit; rollback must not copy tokens
  back to localStorage or introduce a remote/cleartext `server.url`.

## Release decision

**Local security foundation, candidate source and dependency provenance GO; store release NO-GO.** Remaining gates: approved store

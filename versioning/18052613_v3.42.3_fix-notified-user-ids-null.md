# v3.42.3 — Fix: create_open_shift_request null notified_user_ids (error 23502)

**Date:** 2026-05-18
**Branch:** claude/analyze-repo-structure-dE8F4
**Type:** Bug fix (DB function)

## Root cause

`create_open_shift_request` has a two-step write:
1. INSERT the open shift row with `notified_user_ids = ARRAY[v_uid]`
2. After sending notifications, UPDATE `notified_user_ids = (SELECT array_agg(...))`

PostgreSQL's `array_agg()` returns **NULL** (not `'{}'`) when its input is
empty. When 0 notification rows matched the broadcast filter, the UPDATE
set `notified_user_ids = NULL`, violating the NOT NULL constraint →
HTTP 400, PostgreSQL error code 23502.

The empty-match scenario occurs when:
- The only position-matching member is the shift creator (excluded by `user_id <> v_uid`)
- OR all matching members are via `enterprise_member_role_allocations` which
  the old notification WHERE clause did not check

## Fix 1: COALESCE on array_agg

```sql
-- Before (broken):
SET notified_user_ids = (SELECT array_agg(DISTINCT n.user_id) ...)

-- After (fixed):
SET notified_user_ids = COALESCE(
  (SELECT array_agg(DISTINCT n.user_id) ...),
  '{}'::uuid[]
)
```

## Fix 2: Allocation-matched members in notification scope

Extended the notification candidate WHERE clause with:
```sql
OR EXISTS (
  SELECT 1 FROM enterprise_member_role_allocations alloc
  WHERE alloc.membership_id = em.id
    AND alloc.workspace_id = _workspace_id
    AND alloc.business_role = _business_role
)
```

This matches the frontend candidate list behavior introduced in v3.41.7.

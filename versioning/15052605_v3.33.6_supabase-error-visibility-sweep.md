## Implementation details

### Pattern: Supabase error-visibility

All fixes in this release follow the same pattern:

```ts
// Before (silent):
const { data } = await supabase.from('table').select('...');
const items = (data as any[]) || [];

// After (observable):
const { data, error } = await supabase.from('table').select('...');
if (error) { console.error('[Component] fetch failed:', error.message); return; }
const items = (data as any[]) || [];
```

The Supabase JS client returns `{ data, error }` on DB-level failures —
it NEVER throws. A bare `{ data }` destructure silently discards the error,
leaving the caller with `null` data and no observable signal.

### sync-holidays: RPC auth failure classification

Previously: `has_enterprise_role` RPC errors were silently treated as
"not authorized" (403). An operational RPC failure (DB unavailable, function
not found) now returns 500 + an error message instead of a false 403.

### sync-holidays: individual insert failures

Previously: insert errors within the loop were silently swallowed; only
`!insErr` was used to increment the `inserted` counter, masking failures.
Now: each insert failure is logged explicitly; the `else` branch ensures
`inserted` accurately reflects successful inserts only.

### capacityEngine.ts + run-report: hardcoded locale strings

`'Ismeretlen'` (Hungarian: "Unknown") is not a valid English fallback for
an international product. Changed to `'Unknown'` in both locations.

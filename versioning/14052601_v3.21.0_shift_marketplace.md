## What's new

Peer-to-peer shift trading: a member offers their assigned shift,
eligibility-validated colleagues see the offer, first to accept
triggers manager approval (or auto-approves per workspace policy).

The core difference from the existing SubstituteInbox: this is for
**shift assignments** (work the member is scheduled to perform),
not **leave coverage** (work to be performed while the requester is
on approved leave).

## DB structure

| Object | Purpose |
|---|---|
| `shift_trade_offers` | Per-shift offer rows |
| `shift_trade_acceptances` | Per-offer acceptance rows |
| `enterprise_workspaces.shift_trade_auto_approve` | Workspace policy: auto-approve same-skill trades |
| `shift_trade_is_eligible(membership_id, shift_assignment_id)` | STABLE eligibility helper |
| `shift_trade_offer(...)` | SECURITY DEFINER, only the assigned member offers |
| `shift_trade_accept(offer_id)` | SECURITY DEFINER, validates eligibility |
| `shift_trade_decide(acceptance_id, approved, notes)` | SECURITY DEFINER, manager decision; reassigns the shift if approved |
| `shift_trade_cancel(offer_id)` | SECURITY DEFINER, offering member or manager |

All 2 tables policy-block direct INSERT. The 4 RPCs are the sole writers.

## Eligibility rules

A member is eligible to accept a shift trade if:
1. They are active in the same workspace.
2. They are not the offering member.
3. They are not on approved leave that day.
4. They are not already assigned to a shift on the same day.

(Skill matching and hours-budget gating are documented as deferred.)

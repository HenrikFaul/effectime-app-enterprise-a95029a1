# v3.33.0 — Direct user creation + tier-filtered role permissions

**Date:** 2026-05-14
**Branch:** `claude/review-project-requirements-lR0Wp`

Two user-requested capabilities shipped together.

## Task 1 — Direct user creation (no email)

### User intent

> "Not only by invitation should you be able to add a new user, but
> also by user creation, so if I provide the data the same way as in
> invitation then I can also choose 'create user' option, when it
> automatically creates the user, does not send them an email. In this
> case the password is set by me as the creator (so this is a
> difference), which of course they can change later, and password
> creation should have general requirements like at least 10
> characters, at least 1 uppercase lowercase special char number…"

### What ships

**InviteMemberDialog** now has a mode toggle at the top:
- **Invite by email** (existing flow — sends invitation email, user
  sets own password).
- **Create user directly** (new flow — admin sets initial password,
  no email).

In create mode a password section appears with:
- Password input + show/hide toggle.
- Confirm-password input with live mismatch warning.
- Live policy checklist (green check / muted X per rule).

### Password policy

| Rule | Failure code |
|---|---|
| Min 10 characters | `min_length_10` |
| ≥1 uppercase letter | `requires_uppercase` |
| ≥1 lowercase letter | `requires_lowercase` |
| ≥1 digit | `requires_digit` |
| ≥1 special (non-alphanumeric) | `requires_special_char` |

Enforced in two places:
1. **Client-side** via `src/lib/security/passwordPolicy.ts` for live UX.
2. **Server-side** via the Postgres `validate_password_policy()`
   function + the create-workspace-user edge function. Defense in depth.

### Files
- DB migration `v3_33_0_user_create_and_tier_role_perms` (applied to remote).
- `supabase/functions/create-workspace-user/index.ts` (new edge fn, v1 deployed).
- `src/lib/security/passwordPolicy.ts` (new helper).
- `src/components/enterprise/InviteMemberDialog.tsx` (mode toggle + password section + handleCreateUser).

## Task 2 — Tier-filtered role permissions

### User intent

> "In settings under role-permission settings when I assign permissions
> to roles, the assignable permissions should only be those that are
> available/meaningful in the given tier, i.e. what I configure in the
> superadmin menu. Solve so that what functions I make available in
> the tier that the workspace was created in, I can manage those
> permissions for roles."

### The taxonomy challenge

Two different feature catalogs exist:
- `enterprise_feature_catalog` — 23 high-level UI permission slots
  (calendar, reports, members, …) used by the role-permission UI.
- `features` — ~150 fine-grained business feature_keys
  (`ai_copilot_chat`, `access_inbox`, …) used by tier_features.

They don't map 1:1, so a direct filter would erase most permissions.

### Solution

Add a `tier_feature_keys text[]` column to
`enterprise_feature_catalog` that maps each UI permission slot to one
or more `features.feature_key` entries. A UI slot is visible iff:
- Its `tier_feature_keys` array is empty (admin/system permission,
  always visible), OR
- At least one of its mapped tier features is enabled in the
  workspace's active tier.

### Seeded mappings (23 entries)

| UI slot | Mapped tier features |
|---|---|
| `calendar` | calendar_monthly, calendar_filters, annual_view |
| `leave_requests_view` | leave_my_view, leave_team_view |
| `leave_requests_submit` | leave_submit, leave_cancel, leave_half_day |
| `approvals` | approval_inbox, approval_individual, approval_bulk, approval_chain |
| `members` | members_list, members_invite, instant_member_create, invitation_accept, member_edit |
| `rules` | leave_daily_rules, rule_templates, office_coverage_rules |
| `reports` | run_report, scheduled_reports, executive_dashboard |
| `audit` | audit_log, compliance_engine |
| `export` | export_center, csv_import, payroll_export |
| `resources_*` | resource_dashboard, capacity_gap, gantt_timeline, business_roles, skills_mgmt, teams |
| `integrations` | enterprise_workspace_integrations, integration_health |
| ... | ... |

### New RPC

`workspace_permission_catalog(workspace_id)` STABLE SECURITY DEFINER —
returns each catalog row with a `visible boolean` flag computed
against the workspace's active tier_features. Member-only execute.

### Frontend

- `useEnterprisePermissions` calls the new RPC; falls back to the
  legacy direct SELECT if the RPC returns empty (older workspaces
  without a tier_subscription yet).
- `RolePermissionManager` displays an amber notice when ≥1 permission
  slot is hidden:
  > "N permission slots hidden because the features they control are
  > not in this workspace's subscription tier. Upgrade the tier in
  > /superadmin to manage them."

## Manual QA matrix

1. Open InviteMemberDialog → click "Create user directly" tab.
2. Type a password not meeting the policy → checklist shows red Xs;
   submit button stays disabled.
3. Type a valid password (e.g. `Effectime!2026`) → all 5 rules turn
   green; submit enables.
4. Confirm field doesn't match → mismatch warning shows.
5. Submit → new user is created; toast `"<name> created and added to
   the workspace"`; no email sent.
6. Sign in as the new user with the admin-set password → succeeds.
7. From new user's Profile, change password to something else →
   succeeds via existing Supabase auth flow.

8. Settings → Role-permission manager → on a Freemium workspace, see
   amber "N permission slots hidden" notice; locked tier features'
   permission rows are not in the tree.
9. Upgrade the workspace tier in /superadmin to Enterprise →
   permission slots appear, notice disappears.

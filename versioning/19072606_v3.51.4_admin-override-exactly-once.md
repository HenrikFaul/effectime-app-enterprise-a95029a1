## Goal and evidence

The shipped ten-argument `create_admin_leave_override` call had no end-to-end
idempotency identity. If the server committed a request but the client lost the
response, a retry could create another leave request and duplicate approval,
quota and audit effects. A page reload, WebView process restart or concurrent
browser context also had no durable, actor/workspace-bound evidence with which
to reconcile the uncertain outcome.

The delivery therefore adds a versioned v2 path without changing the existing
v1 function contract. Its guarantee is deliberately narrow: calls carrying the
same authenticated actor, workspace, idempotency key and canonical payload
fingerprint replay the same successful receipt. It does not claim global
exactly-once behavior for legacy v1 calls or another device that generated a
different key.

## Implemented contract

- Adds the backward-compatible `create_admin_leave_override_v2` RPC while
  preserving the v1 function OID, definition, owner, execution ACL and runtime
  behavior for existing web and Capacitor clients.
- Adds a private, RLS-enabled ledger keyed by workspace, authenticated actor and
  UUID idempotency key. It stores a canonical SHA-256 payload fingerprint and
  the exact successful response, never the raw justification or comment.
- Serializes the actor and target membership checks and revalidates active
  membership, tenant, role, permission and entitlement state for every new
  operation. Request, approval, quota, audit and ledger writes commit or roll
  back together.
- Same-key/same-payload retries replay the stored receipt. Reusing a key with a
  different payload fails closed. A completed exact replay is resolved before
  mutable authorization state can misclassify the already committed operation.
- Reserves the `leave_request.admin_override` audit action for the trusted
  atomic workflow and prevents direct runtime forgery of that event.
- Routes web, Android and iOS through the same typed Supabase adapter. UUID
  generation uses Web Crypto, the request is bounded to 30 seconds, and malformed
  responses or transport ambiguity fail closed without logging payloads or keys.
- Adds a PII-free retry/reconciliation outbox. Only the version, UUID scope,
  payload digest, UUID key and creation time persist. The seven-day value is a
  reconciliation horizon, not an automatic deletion TTL.
- Allows one unresolved operation per actor/workspace scope. An unchanged
  payload reuses its key; a different payload is blocked until the earlier
  operation is reconciled. Every failure retains evidence, and exact-key cleanup
  is allowed only after a valid server receipt.
- Requires origin-wide Web Locks for the privileged client workflow. Runtimes
  without the API fail before storage access or RPC dispatch. The iOS deployment
  target is 15.4, the first supported WebKit release with that boundary;
  Lockdown Mode remains an explicit fail-closed limitation.
- Introduces no new dependency.

## Compatibility and rollout

The database change is additive. Existing v1 binaries continue to use the
unchanged ten-argument function, but v1 does not receive an idempotency key and
therefore does not gain the v2 guarantee.

Required rollout after migration-history reconciliation:

1. take and verify a production backup and restore it into isolated staging;
2. reconcile the target migration history and review the exact schema diff;
3. apply the v2 migration and verify function OIDs, owners, ACLs, policy/trigger
   catalogs and the PostgREST schema cache;
4. run v1 compatibility and v2 same-key replay/adversarial acceptance;
5. deploy the attestable web/Android/iOS client artifact from the same source
   revision only after the database contract is available;
6. verify the live release identity and record provider deployment IDs.

Deploying the v2-only client before the v2 RPC is visible through PostgREST is
forbidden.

## Rollback and recovery

Before database apply, a normal git revert removes the candidate. After apply,
rollback is client-first: return the client to the compatible v1 path while the
additive v2 function and ledger remain installed. The ledger must not be dropped
while any supported v2 client can retry a recorded key. Data-affecting failure
requires verified backup recovery or a separately reviewed forward repair, not
an ad-hoc destructive down migration.

Uncertain client evidence must be reconciled against the server receipt. It
must not be deleted merely because it is old, malformed or inconvenient to
recover.

## Remaining risks and release decision

- Production migration history remains materially different from the
  repository and a clean migration replay is not available.
- Restored-staging inventory, linked apply and real PostgREST schema-cache
  acceptance have not run.
- Physical Android/iOS transport-loss, process-restart, Web Locks and Lockdown
  Mode behavior remain device acceptance gates.
- A different device can generate a different key; cross-device global
  exactly-once is not claimed.
- The separate global `profiles`/birthday-preferences tenant privacy finding is
  not repaired by v3.51.4.

**Decision:** merged and hosted source candidate **GO**; production database,
web/Edge and store rollout **NO-GO** pending the documented gates.

## Design principles applied

**Repository-aware**: Every prompt file instructs Claude to locate metric definitions in `supabase/migrations/`, source components in `src/components/analytics/`, and version history in `CHANGELOG.md` before producing output. Metric interpretation is grounded in repository evidence, not assumptions.

**Version-aware**: The SYSTEM.md defines a version-induced metric shift protocol. Every anomaly, trend, or KPI variance check starts with a CHANGELOG/migration review before attributing cause to business events.

**Anti-hallucination disciplined**: Explicit prohibition lists in SYSTEM.md Section 8 and each specialist file prevent Claude from stating metric values without source citations, claiming causation without evidence, or presenting low-quality data as reliable analysis.

**Mandatory output contracts**: Every BI output includes three fixed blocks — BI Context (scope, version, source, quality), Analysis, and Confidence Assessment. No output is permitted without all three blocks.

**Effectime-specific**: Prompts reference Effectime's actual metric domains (attendance, compliance, wellbeing, workforce, platform, agile), known data quality patterns (RLS scope, multi-tenancy isolation, soft-delete handling, integration sync gaps, survey response bias), and the full segmentation dimensions (workspace tier, size, industry, locale, role, team, tenure).

- Routing table (BI question type → specialist file)
- Rule precedence (10-level, highest = anti-hallucination)
- Mandatory output contract (3-block structure)
- Anti-hallucination protocol (explicit prohibition list)
- Version-aware reasoning protocol (version-induced shift table)

- Cross-references (8 related repository paths)

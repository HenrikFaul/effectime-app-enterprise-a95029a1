## Problem and evidence

The member sheet formerly saved one user action through up to four independent
requests: membership metadata update, allocation delete, allocation insert and a
self-profile rename. Each request committed separately. An ignored delete error,
network loss or concurrent writer could therefore leave a partial profile, erase
allocations, overwrite a newer value or report success without an auditable
transaction. Related editable data was also loaded from separate snapshots, so
an allocation and its membership metadata could never be proven mutually current.

## Implemented boundary

- `get_workspace_member_profile_edit_snapshot_v1(workspace, membership)` returns
  membership metadata, office, base hours, complete role allocations, opaque
  `profile_revision` and self-only global display name from one SQL statement.
- `save_workspace_member_profile_v1(...)` authorizes and locks actor, target,
  permission, office, profile and allocations, then rechecks entitlement inside
  the same transaction; validates the full immutable draft; commits every change
  and one `membership.profile_updated` audit receipt; or rolls the entire
  operation back.
- Active owner or a role with `members:edit` plus `members_list` may edit
  tenant-owned membership data. Only the target user may update their own global
  display name. Readonly/inactive/cross-tenant callers fail closed.
- `profile_revision` is server managed. The save requires its exact expected
  value; self-name changes additionally compare the exact authoritative display-
  name baseline. Conflict, deadlock and bounded lock timeout require reload.
- The office relation now proves `(office_id, workspace_id)` together. Allocation
  rows prove `(membership_id, workspace_id)` together and restrictive policies
  intersect legacy coarse write policies.
- Existing incompatible lexical, canonical, cardinality, metadata, office and
  display-name rows are inventoried. The migration aborts with counts and never
  trims, normalizes, reassigns or deletes business data silently.
- New saves accept at most 20 NFKC/case-unique roles, two-decimal percentages,
  exactly 100 percent total and exactly one priority role when non-empty. Legacy
  invalid snapshots remain readable so they can be deliberately repaired.
- The shared TypeScript adapter strictly validates UUIDs, exact response keys,
  types, text bounds, decimal precision, revision monotonicity and audit receipt;
  it supports caller cancellation and a 15-second timeout without a raw-table
  fallback. The same adapter is bundled for web, Android and iOS.
- `MemberProfileSheet` fails closed on target-load errors, separates supplemental
  peer data from the authoritative snapshot, suppresses stale/unmounted callbacks
  and duplicate submissions, and requires reload after a conflict. Every runtime
  mount refetches authoritative member data after success.
- The personal Profile page now requires a successful authoritative load, uses
  an exact prior-name `UPDATE` predicate, treats zero rows as a conflict, validates
  names with the shared contract and renders localized load/retry/error states.
- Browser signup, Auth metadata and every inventoried `join-event` display-name
  writer now use the same bounded Unicode contract. The SQL canonicalizer mirrors
  ECMAScript edge whitespace and the auth trigger is recreated exactly; hostile
  disabled-trigger and `service_role` ACL drift are repaired on repeat apply.
- Temporary rename/upgrade no longer races two writable name stores: the profile
  is authoritative until finalization, update/prepare use exact-state CAS and an
  active upgrade freezes rename/reprepare. OAuth finalization uses only that
  frozen name, verifies the Auth Admin update, then runs an idempotently retryable
  auth-first cleanup; uncertain Auth state cannot trigger DB deletion.

## Database compatibility and migration strategy

The migration is forward-only and intentionally strict. It reuses and preserves
the shipped v3.51.5 self-name RPC signature while upgrading its locking/revision
behavior. Candidate read/save RPCs are additive, but a client that calls them
before PostgREST exposes the migrated schema fails closed. Rollout order:

1. verified backup and isolated restore;
2. migration-history/content reconciliation and 128/128 clean replay;
3. restored-production-like inventory with approved data-preserving remediation;
4. apply database migration and reload/verify schema cache;
5. verify exact columns, constraints, policies, owners and ACLs;
6. immediately publish the same attestable web/Android/iOS client;
7. run two-tenant, multi-role and concurrent old/new-client acceptance.

Rollback hides or reverts the client editor first. Tenant/FK/audit/lexical guards
must not be loosened to restore an older UI. Any constraint change requires a new
reviewed forward migration.

## Security and privacy notes

Audit metadata records changed field names and allocation counts, not display
names, free text, role names, locations or before/after snapshots. RPC errors are
bounded in the client and raw database details are not rendered. No runtime
dependency, secret, privileged mobile key or alternate data store was added.

# v3.39.0 — Smart Staffing Workflow

**Released:** 2026-05-16
**Branch:** claude/analyze-repo-structure-dE8F4
**Scope:** Supabase DB + backend RPCs, coverageEligibility engine, CoveragePlannerView, EmployeeDashboard, 8 locale files

---

## Problem solved

Managers at multi-location retail/optical businesses were managing shift coverage via Excel files and email threads. Employees had no digital way to signal availability; managers had no visibility into who was free before assigning; and unfilled slots had no broadcast mechanism that could fill themselves through employee self-service.

## What was built (smallest possible surface)

### Database (1 migration)
- `enterprise_staff_availability` — workspace-scoped employee availability rows (available / preferred / unavailable per date). Replaces the personal-only `personal_availability` table for manager-visible workflows. RLS: members manage own rows; managers read all.
- `enterprise_open_shift_requests` — manager-posted unfilled slots with office, date, role, notes, and fill status. RLS: all members can read; managers can insert.
- `enterprise_open_shift_claims` — employee claim attempts. UNIQUE(request_id, user_id) prevents double-claiming.

### RPCs
- `claim_open_shift(_request_id)` — SECURITY DEFINER, uses `SELECT … FOR UPDATE` row lock to prevent race conditions. Atomically: validates status=open, inserts claim, creates shift assignment, marks request filled, supersedes other claims, fires `shift_assigned` notification to claimant and `open_shift_filled` to managers.
- `create_open_shift_request(...)` — SECURITY DEFINER, admin/owner only. Creates the request and broadcasts `open_shift_broadcast` notification to all active members.

### Engine extension
- `EligibilityContext.availabilityByDate` — optional `Map<string, Set<string>>` (date → user_id set). When present, `evaluateEligibility()` gives +20 score boost to self-marked-available members. Fully backward-compatible (undefined = no change).

### UI — Manager
- `CoveragePlannerView` now loads `enterprise_staff_availability` in the same `Promise.all` as existing data sources. Green dot indicator next to candidate names who are self-marked available on the target date. New "Open shift broadcast" section at the bottom of each cell drawer via `OpenShiftManager`.

### UI — Employee (EmployeeDashboard)
- `AvailabilityCalendar` — monthly calendar with tap-to-cycle (→ available → preferred → unavailable → clear). Upserts via `enterprise_staff_availability`. Shown above the existing ShiftMarketplacePanel.
- `OpenShiftPanel` — lists all open shifts for the workspace; first-come-first-served claim with confirmation dialog. Graceful error messages for already-filled and race-condition cases.

### Notifications (reuses existing infrastructure)
Four new `event_type` values on `enterprise_notifications`:
- `shift_assigned` — claimant confirmation
- `open_shift_filled` — manager alert
- `open_shift_broadcast` — member notification for new open shift

### Tests
- `src/test/smartStaffing.test.ts` — 7 tests covering: availability boost (+20), no boost for other users, no boost for different dates, backward compatibility without field, ranking with availability, ranking precedence (blocking leave beats availability boost).

### i18n
- 2 new namespaces (`availability`, `open_shifts`) + 3 new `coverage_planner` keys added to **all 8 locale files** (en, hu, de, at, cs, sk, pl, ro).

## What was NOT built (deliberately excluded)
- Excel-like grid UI
- New scheduling screen (extends existing CoveragePlannerView only)
- Changes to `personal_availability` (left untouched)
- Duplicate models for anything already in enterprise_shift_assignments
- Background jobs (notifications fire synchronously inside RPCs)

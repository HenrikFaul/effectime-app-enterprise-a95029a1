# v3.33.4 — Password-policy parity + audit-log error-visibility fix

**Date:** 2026-05-15
**Branch:** claude/architect-team-session-UWfpK
**Type:** Bug fix (no new features)

---

## Goals

Close two correctness gaps found during a full-system codebase audit:

1. **Password policy split** — `ChangePasswordCard` enforced an 8-character
   minimum while the authoritative server-side `validate_password_policy()`
   (Postgres function, introduced in v3.33.0) and the `create-workspace-user`
   edge function both require 10 characters. Users could set a password that
   the UI accepted as valid but that violated the company policy.

2. **Silent audit-log failures** — `logAuditEvent()` used a bare `await` on
   the Supabase insert. The Supabase JS client never throws on DB-level errors;
   it returns `{ data, error }`. The `try/catch` therefore never fired for
   actual insert failures (RLS rejection, constraint violation), so audit events
   were lost silently without any observable signal.

## Implementation details

### Fix 1 — Password policy parity

The v3.33.0 release introduced `validate_password_policy()` (Postgres IMMUTABLE
function) and `src/lib/security/passwordPolicy.ts` (frontend mirror), both
requiring ≥ 10 characters. The `InviteMemberDialog` (mode = 'create') already
used the correct 10-char implementation.

`ChangePasswordCard` and `PasswordRequirements` consumed the older
`src/lib/passwordValidation.ts` with `minLength: password.length >= 8`.
The `PasswordRequirements` UI would show a green checkmark for an 8- or 9-char
password, misleading users into thinking their new password was policy-compliant.

No migration of existing passwords is needed — Supabase Auth has no custom
password policy enforced at the DB level; this fix only closes the UX gap.
The two implementations (`passwordValidation.ts` and `passwordPolicy.ts`) now
agree on the threshold. A future cleanup could consolidate them into a single
module (tracked as open question).

### Fix 2 — Audit log error visibility

```ts
// Before (silent on Supabase errors):
await supabase.from('enterprise_audit_events').insert([...]);

// After (explicit error check + return signal):
const { error } = await supabase.from('enterprise_audit_events').insert([...]);
if (error) { console.warn('Audit log insert failed:', error); return false; }
return true;
```

`logAuditEvent` now returns `Promise<boolean>`. Existing callers that discard
the return value are unaffected; new callers can react to failures if needed.

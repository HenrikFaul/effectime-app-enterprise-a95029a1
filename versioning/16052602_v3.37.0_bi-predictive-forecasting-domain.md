## Summary

Adds a complete predictive analytics and forecasting domain to the Business Intelligence prompt architecture. Eight new prompt files and one new template establish a rigorous, constraint-governed forecasting system for Effectime's workforce metrics. This is Phase 3 of the BI framework (Phase 1: v3.35.0, Phase 2: v3.36.0).

## New files

### Core forecasting framework (2 files)

**`prompts/forecasting_methodology.md`** (master forecasting controller)
Defines the six permissible forecasting methods (trend extrapolation, seasonal projection, moving average, leading indicator, driver-based, scenario), the maximum reliable horizon for each, six non-negotiable forecasting responsibility rules (horizon discipline, confidence interval requirement, training window version integrity, assumption transparency, forecast decay declaration, decision-purpose requirement), the pre-forecast checklist, the domain routing table for all predictive tasks, version-aware forecasting protocol, and the Forecast Master Block output format that every domain forecast must embed.

**`prompts/seasonal_pattern_library.md`** (reference file for all forecasting prompts)
Documents 16 named seasonal patterns across five metric domains: attendance and time (AT-01 through AT-04), compliance (CP-01 through CP-03), engagement and wellbeing (EW-01 through EW-03), financial and labor cost (FC-01 through FC-03), and platform and adoption (PA-01 through PA-02). Each pattern includes: period table with seasonal indices, evidence basis, jurisdiction applicability (EU / UK / cross-market), minimum data requirements to validate applicability, and the protocol for adding new patterns. Patterns are cited by name in all other forecasting prompts.

### Domain-specific forecasting prompts (6 files)

**`prompts/workforce_demand_forecasting.md`**
Four forecasting methods by workspace maturity (D-1: historical average with seasonal, D-2: trend-adjusted, D-3: compliance-floor anchored, D-4: event-adjusted). Demand driver identification by feature availability and tier. Short/medium/long-term horizon guidance with widening uncertainty rules. Demand-to-headcount FTE conversion formula. Forecast validation protocol with backtesting requirement. Output format with per-period projection table and headcount gap computation.

**`prompts/turnover_risk_scoring.md`**
Nine behavioural signals from Effectime data with weight (high/medium/low) and direction. PII and privacy constraints as non-negotiable Section 2 (individual-level authorization, minimum team size N ≥ 5, no named employees in output, score decay at 4 weeks). Risk scoring matrix producing High/Medium/Low classification from signal combinations with tenure-window elevation rule. Team risk classification (High + Medium > 40% = team-level flag). Turnover volume projection formula with industry-average multipliers and explicit ± 50% uncertainty. Historical validation protocol.

**`prompts/wellbeing_trajectory_forecasting.md`**
Data sufficiency check (response rate ≥ 40%, minimum 3 survey cycles). Seasonal adjustment using Pattern EW-01 to separate seasonal from operational decline. Trend velocity classification (slow/moderate/rapid/improving). Time-to-threshold projection formula. Proxy signal adjustment for inter-survey periods (4 signals: voluntary overtime acceptance, absence rate, punch corrections, shift cancellations). Signal-based estimate protocol when data is insufficient. Burnout risk tier classification (Critical/High/Elevated/Monitoring) with recovery timeline estimates. Team risk matrix.

**`prompts/financial_forecasting.md`**
Labor cost projection protocol (5 steps: cost base, scheduled cost, actual cost with variance rate, overtime cost with seasonal adjustment, budget variance trajectory). Budget burn rate index with escalation thresholds. Overtime driver classification (structural/seasonal/absence-driven/scheduling/compliance) with per-driver intervention savings computation. Payroll correction cost trend with admin cost formula. Marketplace fill cost projection. Horizon and uncertainty table by cost component. Version check for financial metric computation changes.

**`prompts/scenario_modeling.md`**
Scenario definition protocol (4 steps: decision, baseline, alternatives, comparison metrics). Impact modeling chains for five decision types: staffing level change (FTE calculation with onboarding ramp), marketplace incentive rate change (acceptance rate curve approximation), compliance rule configuration change (knowledge lag model), shift schedule restructure (cascade to wellbeing), tier upgrade (adoption curve modeling). Scenario comparison table structure. Uncertainty stacking warning for multi-step assumptions. Decision recommendation framework (dominant/risk-adjusted/minimum viable). Output format with assumptions registry.

**`prompts/capacity_demand_gap_forecasting.md`**
The highest-value predictive output: combines demand-side projection with supply-side trajectory. Supply trajectory modeling (confirmed hires with onboarding ramp, confirmed departures, planned leave, predicted absence with seasonal adjustment). Gap computation formula per period with four-tier classification (surplus/at-risk/gap/critical). Gap inflection point identification (decision trigger). Intervention lead time analysis by intervention type (marketplace 1–2 weeks, schedule 2–4 weeks, temporary cover 1–3 weeks, permanent hire 6–12 weeks). Three-scenario uncertainty projection (pessimistic/central/optimistic). Cascade effects chain (gap → overtime → WTD breach → wellbeing → turnover → worsening gap). Compliance floor breach detection (regulatory event trigger).

## Design principles for the predictive domain

**Anti-extrapolation discipline**: Every forecasting method has a documented maximum horizon. Exceeding it requires an explicit "speculative" label. No numeric forecast is permitted with fewer than 6 data points.

**Version-integrity requirement**: Training windows may not span CHANGELOG version breaks that altered the metric's computation. Truncation is mandatory; the truncated window must still meet minimum data requirements.

**Assumption registry**: Every driver-based and scenario forecast lists all assumptions with source and sensitivity. "No assumptions" is only permitted for pure trend extrapolation on verified data.

**Decay dates**: Every forecast has a stated validity date after which it must not be used for decisions without re-running.

**PII-first design**: The turnover risk scoring prompt places PII constraints as Section 2 (before the scoring protocol) to prevent inadvertent individual disclosure before the constraint is read.

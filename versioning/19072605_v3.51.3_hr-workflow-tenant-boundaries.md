# v3.51.3 — HR workflow tenant-boundary repair candidate

**Date:** 2026-07-19

**Branch:** `codex/hr-workflow-tenant-boundaries`

**Release state:** source candidate; production database/Edge/web rollout is
**NO-GO** until the linked migration history and restored-staging inventory are
reconciled.

## Evidence and requirement

The historical HR workflow schema and RPCs correlate membership, assignee and
parent IDs globally instead of by workspace. The reproduced impact includes a
cross-workspace assignee update path, member/admin RLS disclosure risk, hidden
legitimate delegated tasks, and cross-tenant data loss through a legacy global
`ON DELETE CASCADE` parent relationship.

## Implemented repair

- Adds the forward-only
  `20260719000000_v3_51_3_hr_workflow_tenant_boundaries.sql` migration. No
  historical migration or existing row is rewritten, deleted or reparented.
- Emits aggregate-only legacy mismatch/inactive-reference inventory.
- Adds locked trigger validation for active, same-workspace templates,
  memberships, instances and task assignees. Parent workspace identifiers are
  immutable while referenced.
- Blocks template, membership and instance parent operations when a historical
  global FK would cascade or null a row belonging to another workspace; valid
  same-workspace FK behavior remains unchanged.
- Correlates instance/task RLS by workspace and active membership. A private,
  non-PostgREST `SECURITY DEFINER` predicate preserves the documented delegated-
  assignee read path without exposing parent-instance RLS data.
- Replaces all four public HR RPCs with their existing names, named arguments,
  defaults and return schemas, plus row locks, tenant validation, status
  allowlists, idempotent same-state retries, trusted `search_path=pg_catalog`
  and exact execution ACLs. Only `authenticated` may execute these user-context
  RPCs; external service-role callers must migrate to an authenticated user
  context rather than impersonating one.
- Separates admin/member inbox data paths in the web client, omits the status
  predicate for the `all` filter, and renders retryable instance/task errors
  instead of false empty states.

## Compatibility and rollout

The public RPC input names/types/defaults and list return order remain
unchanged. Existing FK names/actions remain available to PostgREST; guards only
fail closed for tenant-inconsistent legacy relations. The web change preserves
admin and member behavior while removing intentional 42501 fallback traffic.

Required rollout after history reconciliation:

1. verified backup and isolated restore;
2. aggregate inventory on restored staging and reviewed, data-preserving
   remediation for every non-zero cross-workspace count;
3. clean 125/125 replay, schema fingerprint/types diff and zero database lint
   errors;
4. apply database migration, then same-SHA Edge and web artifacts;
5. run the targeted contract plus two-tenant authenticated UI/API smoke and the
   live release-identity verifier.

## Rollback and recovery

Before database apply, normal git revert removes this candidate. After apply,
do not use a destructive down migration: triggers, policies, RPCs and private
helper are security boundaries, and removing them would reopen the issue.
Rollback requires a separately reviewed forward repair after restoring from the
verified backup if data integrity is affected. The frontend unit can be reverted
independently, but doing so would restore false-empty member error behavior.

# v3.16.0 — Workspace UUID in URL + Back-button regression fix

**Date:** 2026-05-13
**Branch:** `claude/review-project-requirements-lR0Wp`

## Problem statement

Two issues raised together by the user during testing:

1. **Workspace identity wasn't bookmarkable.** Every workspace-scoped
   page in the app rendered at `/app?tab=<name>` regardless of which
   workspace the user was in. The active workspace lived in
   `localStorage` + `?ws=<uuid>` query param. You couldn't send a
   colleague a link to a specific workspace's calendar — they'd land
   wherever their own localStorage had pointed them last.

2. **Back button dropped to the landing page.** After picking a
   workspace from the picker grid, pressing the browser Back button
   from any dashboard tab skipped the picker entirely and dumped the
   user onto `/`.

## What changed

### New routing principle (non-negotiable from v3.16.0 onward)

```
/                       Landing
/auth                   Public auth
/app                    Workspace picker
/app?select=1           Picker (force-show, skip auto-redirect)
/app?invite=<token>     Picker (invite acceptance handler)
/w/<workspaceId>        Dashboard root (default tab: members)
/w/<workspaceId>?tab=X  Dashboard with tab X selected
/profile                Profile (non-workspace)
/admin                  Admin (non-workspace)
/superadmin             Superadmin (non-workspace)
```

The workspace UUID is now a path segment. Tabs remain query params
(intentional — keeps tab-history pushState behavior simple).

### Governance rule update

`.governance/ui_ux_rules.md` now contains two relevant principles:

- **"Browser Back button must always work"** (existed since v3.7.2;
  expanded to enumerate the exact cases where `replace: true` is
  permitted: consumed tokens, auth/permission redirects, legacy-URL
  migration shims, transient same-tab auto-redirects).
- **"Workspace identifier in URL"** (new in v3.16.0; codifies the
  workspace-UUID-in-path rule, the deep-link rationale, the
  back-button consequence, and the legacy URL compatibility shim).

`CLAUDE.md` quick-reference references both.

### Root cause of the back-button bug

`src/pages/Enterprise.tsx` previously held the active workspace in
`useState` + `localStorage`. When the user clicked a workspace card,
the code called:

```ts
setSearchParams(nextParams, { replace: true });  // line 98 (old)
```

`replace: true` overwrites the current history entry instead of
pushing a new one. So a user trajectory `/` → `/app?select=1` →
(click card) → `/app?tab=members` had ONLY TWO history entries:
`/` and `/app?tab=members`. The picker step was invisible. Pressing
Back from any tab fell straight through to `/`.

The same pattern repeated at line 187 (auto-select for returning
users with `?ws=<uuid>` in the URL).

### Fix

`Enterprise.tsx` is now URL-param-driven:

```tsx
const { workspaceId: workspaceIdFromUrl } = useParams<{ workspaceId?: string }>();
const selectedWorkspaceId = workspaceIdFromUrl ?? null;

const pickWorkspace = (id: string) => {
  localStorage.setItem(ACTIVE_WORKSPACE_KEY, id);
  navigate(`/w/${id}`);   // push, NOT replace — this is the fix
};
```

The picker step is now a real history entry. Back returns to it.

### Auto-redirect for returning users

On `/app`, if the user has a valid last-used workspace and didn't pass
`?select=1`, the page replace-redirects to `/w/<lastUsed>`:

```tsx
useEffect(() => {
  if (workspaceIdFromUrl) return;
  if (forceSelector) return;
  if (inviteToken) return;
  if (loading || acceptingInvite) return;
  if (legacyWsParam) return;
  if (workspaces.length === 0) return;

  const stored = localStorage.getItem(ACTIVE_WORKSPACE_KEY);
  const candidate = (stored && workspaces.find((w) => w.id === stored)) || workspaces[0];
  if (candidate) {
    localStorage.setItem(ACTIVE_WORKSPACE_KEY, candidate.id);
    navigate(`/w/${candidate.id}`, { replace: true });
  }
}, [...]);
```

`replace: true` is correct here per the updated governance rule: the
source URL (`/app` with no params) is transient and has no meaning to
the user — they didn't intend to dwell on it.

### Legacy URL compatibility

Old bookmarks `/app?ws=<uuid>` are auto-redirected (replace) to
`/w/<uuid>` preserving other query params. Existing browser bookmarks
keep working.

### Invalid deep-link handling

If the user follows `/w/<uuid>` for a workspace they no longer
belong to (kicked out, workspace deleted), a toast fires
("Workspace not found or you are no longer a member. Pick another
one.") and they're replace-redirected to `/app?select=1`.

### Invite-acceptance flow

`/app?invite=<token>` still works. After successful acceptance, the
flow now `navigate`s (replace) to `/w/<new_workspace_id>` instead of
mutating searchParams in place.

## Out of scope (deliberate non-changes)

- **Tabs stay as query params** (`?tab=calendar`), NOT path segments
  (`/w/<id>/calendar`). Migrating tabs to path segments would force
  every dashboard sub-component to learn nested routing — invasive,
  zero benefit beyond aesthetics.
- **No `useActiveWorkspace()` hook** introduced. The workspace is still
  passed as a prop through the dashboard tree. A future refactor can
  add the hook as a clean-up; this PR keeps the diff focused on URL
  shape + back-button.
- **`/profile`, `/admin`, `/superadmin` remain non-workspace routes**.
  They are workspace-agnostic by design (account-level).

## What v3.20.0 delivers

1. **Per-workspace jurisdiction selector** — workspace owners pick
   EU_WTD (default), HU_MT, DE_ArbZG, AT_AVRAG, or custom. Parameters
   like `weekly_max_hours` and `daily_rest_min_hours` can be overridden
   per-workspace.

2. **WTD weekly-max-hours pre-check engine** —
   `compliance_check_working_time(workspace_id, period_start, period_end)`
   computes approximate weekly hours per member from attendance segments,
   evaluates against the active ruleset, and inserts:
   - **`violation`** rows for hard breaches (>limit)
   - **`warning`** rows for breaches imminent (>90% of limit)

3. **Real-time compliance dashboard** — green/yellow/red status header
   + KPI cards (violations, warnings, status) + period date pickers
   + run button + unresolved findings list color-coded by severity.

4. **Audit-ready violation log** — every check writes immutable rows
   to `compliance_violations`. Owner/RA + platform admin read access.

## Architecture decisions

### Hours computation
Weekly hours are approximated from `enterprise_attendance_segments` —
the sum of segment durations where `segment_type IN ('regular', 'overtime')`
divided by the number of weeks in the requested period. This is a
*reasonable proxy*; a more precise per-week sliding window would
require deeper attendance-period semantics and is out of scope for v3.20.0.

### Warning band at 90%
The strategy doc calls for hard violations only. We added a 10% warning
band proactively so workspaces can see members approaching their limit
before they breach it. This matches DE/AT employment-law operator
practice (give employees and managers notice before a hard breach).

### No pre-publish blocking yet
The strategy doc envisions blocking a schedule publish when violations
exist. v3.20.0 ships the *detection* engine; the *blocking integration*
into `join-event` (or wherever schedule-publish lives) is the next
polish PR. This staged approach lets workspaces evaluate compliance
data on real workloads before turning on the hard block.

## Tier mapping

3 feature_keys (`compliance_engine`, `compliance_dashboard`,
`compliance_export`) mapped to **Pro + Enterprise** only.

**Freemium intentionally excluded** — compliance is the explicit
enterprise-sales unlock per the strategy doc; making it free
dilutes the upsell rationale and undermines the regulatory value-add.

## RLS posture

- `compliance_rulesets`: read = workspace member or platform admin; write = workspace owner/RA only.
- `compliance_violations`: read = workspace owner/RA or platform admin; direct INSERT policy-blocked (only the SECURITY DEFINER RPC writes).

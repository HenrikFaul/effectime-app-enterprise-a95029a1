# v3.17.1 — STRICT tier_key: silent freemium fallback eliminated

**Date:** 2026-05-13 (hotfix follow-up to v3.17.0)
**Branch:** `claude/review-project-requirements-lR0Wp`

## What happened

After v3.17.0 made the tier visible everywhere, the user took a screenshot
showing: dialog with **Enterprise** selected → workspace created → badge
on picker card and dashboard header reads **FREEMIUM**. The user's
intent was lost between the dropdown and the database row.

## What I found in the DB

User's recent create-workspace attempts in a 3-minute window:

| Time | Created tier | Expected |
|---|---|---|
| 20:50:38 | enterprise ✓ | enterprise |
| 20:51:20 | freemium ✗ | (likely enterprise) |
| 20:51:58 | freemium ✗ | (likely enterprise) |
| 20:52:28 | freemium ✗ | (likely enterprise) |
| 20:53:17 | freemium ✗ | enterprise (the screenshot one) |

Same user, same dialog, same browser session. Sometimes the tier
arrived correctly, sometimes it didn't. Highly suggestive of a
parameter-binding race, a stale React closure, or some other transient
path where `_tier_key` arrived NULL/empty at the RPC.

## The actual root cause

The DB function was *forgiving*:

```sql
CREATE OR REPLACE FUNCTION public.create_workspace_with_owner(
  _name text,
  _description text DEFAULT NULL,
  _tier_key text DEFAULT 'freemium',  -- silent default
  _seats integer DEFAULT 5
) RETURNS uuid …
BEGIN
  …
  SELECT id INTO _tier_id FROM public.tiers WHERE tier_key = COALESCE(_tier_key, 'freemium') LIMIT 1;
  IF _tier_id IS NULL THEN
    SELECT id INTO _tier_id FROM public.tiers ORDER BY sort_order LIMIT 1;  -- still freemium
  END IF;
  …
```

If `_tier_key` arrived NULL/empty/unknown for any reason — bad PostgREST
parameter binding, a stale React Select closure, a network truncation,
a server-side coercion — the function silently picked **freemium**. The
user got a workspace, but on the wrong tier, with NO error feedback.

This is the classic "forgiving server" antipattern. Forgiving servers
make bugs invisible by absorbing them.

## The fix — strict server contract + client verification

### DB (applied via Supabase MCP migration `create_workspace_with_owner_strict_tier`)

```sql
CREATE OR REPLACE FUNCTION public.create_workspace_with_owner(
  _name text,
  _description text DEFAULT NULL,
  _tier_key text DEFAULT NULL,   -- no silent default
  _seats integer DEFAULT 5
) …
BEGIN
  _user_id := auth.uid();
  IF _user_id IS NULL THEN RAISE EXCEPTION 'Not authenticated'; END IF;

  _normalized_tier_key := lower(trim(coalesce(_tier_key, '')));
  IF _normalized_tier_key = '' THEN
    RAISE EXCEPTION 'tier_key required: pass one of (%) — got NULL or empty',
      (SELECT string_agg(tier_key, ', ' ORDER BY sort_order) FROM public.tiers);
  END IF;

  SELECT id INTO _tier_id FROM public.tiers WHERE tier_key = _normalized_tier_key LIMIT 1;
  IF _tier_id IS NULL THEN
    RAISE EXCEPTION 'Unknown tier_key: % — pass one of (%)',
      _normalized_tier_key,
      (SELECT string_agg(tier_key, ', ' ORDER BY sort_order) FROM public.tiers);
  END IF;

  -- … then INSERT workspace + tenant + tenant_workspaces + tenant_subscriptions
  -- with metadata.{resolved_tier_key, input_tier_key} for forensic audit.
END
```

No silent freemium ever again. If the dialog sends NULL/empty/unknown,
the function raises and the call fails atomically — no half-created
workspace.

### Client (CreateWorkspaceDialog.tsx)

After both create paths — direct RPC and seed-demo-workspace — the dialog
now reads back the actual tier:

```tsx
const { data: tierRow } = await supabase
  .from('workspace_active_tier')
  .select('tier_key')
  .eq('workspace_id', newWorkspaceId)
  .maybeSingle();
const actualTier = tierRow?.tier_key;
if (actualTier && actualTier !== tierKey) {
  toast.error(t('create_workspace.tier_mismatch_error',
    { requested: tierKey, actual: actualTier }));
}
```

So even if a future RPC regression somehow returns a wrong-tier workspace,
the user gets an IMMEDIATE warning in the same request cycle — not days
later when a paid feature mysteriously doesn't work.

Also added a defensive pre-check: if `tierKey` is empty in the dialog
state (shouldn't happen given the Select is required, but belt-and-braces),
show a localized "please pick a tier" error before the RPC call.

## Localization

2 new keys in EN + HU:
- `create_workspace.tier_required` — "Please pick a subscription plan before creating the workspace."
- `create_workspace.tier_mismatch_error` — "WARNING: workspace was created with the wrong tier (you picked X, got Y). Please contact a platform admin to fix this."

## Backwards compatibility

The function signature default changed from `'freemium'` to `NULL`. Any
caller that omits `_tier_key` will now fail. Audit of current callers:

- `CreateWorkspaceDialog.handleCreate` — sends `_tier_key: tierKey`. ✓
- `CreateWorkspaceDialog.handleCreateDemo` → `seed-demo-workspace` →
  `userClient.rpc('create_workspace_with_owner', { _tier_key: tierKey })`. ✓
- DB-side callers — none (verified via the v3.15.4 multi-method audit).

No live consumer breaks. New callers added in the future will fail fast
on missing `_tier_key`, which is the desired behavior.

## What this DOESN'T do

- It does NOT explain *why* `_tier_key` was sometimes lost between the
  dialog and the RPC. The strictness fix prevents the user from ever
  being affected by that pathology again (they'll get a clear error
  instead of a silent downgrade), but the underlying race remains
  unidentified.
- It does NOT retroactively repair the 4 workspaces the user created
  on the wrong tier earlier today. Those need to be fixed manually via
  the Superadmin → Workspaces → "Change tier…" action shipped in
  v3.17.0. Each fix writes a `platform_audit_events` row.

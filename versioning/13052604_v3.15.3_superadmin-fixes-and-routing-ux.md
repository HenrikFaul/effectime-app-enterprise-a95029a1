# v3.15.3 — Superadmin Platform Control Plane: 3 bugfixes + 2 routing UX additions

**Date:** 2026-05-13
**Branch:** `claude/review-project-requirements-lR0Wp` (PR #73)
**Scope:** Restore the Superadmin Platform Control Plane to a working
state by fixing 3 regressions (Overview cards, Workspaces table, Audit
log 404). Add 2 UX improvements to the Feature & Tier → Routing tab
(click-to-expand-all + flat-path view mode).

## Bugs (3) — Superadmin UI

### Bug 1: Overview tab cards all rendered `—`

**Symptom:** Every stat card (Összes munkaterület, Aktív, Archivált,
Helyreállítás módban, Összes felhasználó, Új 30 nap, Aktív jelzők,
Függő email) showed `—` even though the workspaces table had 13 rows.

**Root cause:** `OverviewTab` typed the response as `PlatformOverview`
with flat fields (`total_workspaces`, `active_workspaces`,
`email_queue_pending`, …), but the `superadmin-hub` edge function
returns:
```json
{
  "workspaces": { "total": 13, "active": 13, "archived": 0, "recovery_mode": 0, "new_30d": 0 },
  "users":      { "total_auth": 564, "total_profiles": 564, "total_members": 200, "new_30d": 4 },
  "features":   { "flags_enabled": 7 },
  "email_queue":{ "pending": 0, "sent": 0, "failed": 0 }
}
```
Every `data.<flat>` read yielded `undefined` → cards rendered the
`value ?? '—'` fallback for every metric.

**Fix:** Inside `OverviewTab.load()` (SuperadminControlPlane.tsx:222),
flatten the nested response to the `PlatformOverview` shape before
`setData`. Edge function output is unchanged (its nested shape is
semantically clearer and may be consumed elsewhere later).

### Bug 2: Workspaces tab rows had no name, locale, timezone, member count

**Symptom:** Workspace table showed dashes for name/locale/timezone,
`23` members in every row (because some shared field happened to match
that index), and the status badge was "Helyreállítás" for every
workspace.

**Root cause:** `WorkspacesTab.load()` cast the response as
`(res as { workspaces: Workspace[] }).workspaces`, but the actual
shape is:
```json
{ "workspaces": [{
    "workspace": { "id": "...", "name": "...", "locale": "...", "timezone": "...", "is_archived": false, "recovery_mode": false, "created_at": "..." },
    "member_count": 23,
    "owner_email": "..."
}, ...] }
```
Every direct read (`w.name`, `w.locale`, `w.timezone`, `w.status`,
`w.created_at`) was `undefined`. `w.member_count` was actually
populated at the top level of the wrapper so it rendered — that's why
the screenshot showed `23` and `25` member counts but everything else
blank. `w.status` being `undefined` made the `statusBadge()` helper
fall through to the orange "Helyreállítás" default — explaining why
every row got that badge.

**Fix:** Added an explicit mapper inside `WorkspacesTab.load()` that:
- Unwraps `w.workspace.{name, locale, timezone, created_at, is_archived, recovery_mode}` to the top level.
- Derives `status`: `recovery_mode ? 'recovery' : (is_archived ? 'archived' : 'active')`.
- Keeps `member_count` from the wrapper unchanged.

### Bug 3: Audit log → `Az audit események betöltése nem sikerült: [object Object]` + network 404

**Symptom:** Audit log tab loaded the table layout, then immediately
showed a red banner with the message `[object Object]`. DevTools
network panel showed a 404 on `platform_audit_events*`.

**Root causes (two stacked issues):**
1. The v3.15.0 release added migration
   `20260513120000_platform_audit_events.sql` to create the
   `platform_audit_events` table, but it had never been applied to the
   remote DB. PostgREST returns 404 for queries against tables it
   doesn't know about. Same for
   `20260513120100_fix_feature_dependencies.sql`, which was also
   missing (caught while listing migrations).
2. The catch in `PlatformAuditLogTab.load()` used:
   ```ts
   const msg = e instanceof Error ? e.message : String(e);
   ```
   PostgrestError is a plain object (not an Error instance), and
   `String({...})` produces `"[object Object]"`. So even when the
   server returned a perfectly readable error like
   `{ message: 'relation "platform_audit_events" does not exist', code: '42P01' }`,
   the UI showed `[object Object]`.

**Fixes:**
- Applied both missing migrations to remote via Supabase MCP
  `apply_migration` (`platform_audit_events` table + RLS + indexes;
  `fix_feature_dependencies` row updates).
- Replaced the catch block in `PlatformAuditLogTab.load()` with an
  explicit extractor that tries `e.message`, then `details`, `hint`,
  `code`, then falls back to `JSON.stringify(e)`. So a future
  PostgrestError surfaces a useful message instead of `[object Object]`.

## Features (2) — Feature & Tier → Routing tab UX

### Click-to-expand-all under a branch

**Before:** Each parent node in the routing tree showed a count badge
(e.g. `3` on `/app/workflows/access`). To see every feature underneath
you had to expand each menu segment manually — sometimes 3 levels
deep, sometimes more.

**After:** The count badge is now an interactive button. One click on
the `3` opens the parent node *and* every descendant node beneath it
in a single state update, so the entire branch becomes visible at
once. Implementation:
- `RoutingTree` collects `collectSubtreePaths(node, path)` (recursive
  walk, returns all `page::…|menu::…|menu::…` keys).
- New outer-component callback `expandSubtree(paths: string[])` writes
  them all to `openPaths` in a single `setOpenPaths` batch and
  persists to the existing localStorage entry.
- Badge click uses `e.stopPropagation()` + `e.preventDefault()` so the
  collapsible's own toggle doesn't fire.

Tooltip on hover: `"Kattints ide az ág összes elemének kibontásához"`.

### Flat-path display mode

**Before:** Only the nested tree view was available. Some operators
(particularly Sales / Customer Success) wanted a flat list view they
could scan or copy as a feature inventory.

**After:** A new view-mode toggle ("Fa nézet" / "Lapos útvonal") sits
at the right edge of the routing tab toolbar. In Flat mode the tree
is replaced by a single sorted list, one feature per line, formatted as:

```
/app/workflows/access / Workflow / Hozzáférés / Inbox / Hozzáférés inbox  [access_inbox] [workflows] 👁 ✏️
```

Sort order: route_path → menu_path → sort_order → feature_key. The
audit banner, filter bar, search, module filter, and tier filter all
work identically to the tree view. View-mode choice is persisted per
user in localStorage (`routingTreeViewMode:v1:<userId>`).

The `RoutingFlatList` component is a sibling of `RoutingTree`; the
parent tab swaps between them based on `routingViewMode` state.

## Known limitations

- The Workspaces tab `name` column shows only the workspace name; the
  edge function returns `description` and `owner_email` but the
  current table layout doesn't display them. Worth surfacing in a
  future revision but out of scope for this release.
- The Flat view doesn't support inline route editing (the editor lives
  in the tree view's pencil button). Editing still works — clicking the
  pencil in Flat mode switches back to tree view with the row in edit
  state — but a future polish pass could surface the editor inline.

**Date:** 2026-06-19
**Branch:** claude/embed-wizard-modal-calendar
**Type:** Feature + Localization bugfix

## Goals

Close two parity gaps the customer reported on the guest embed vs. the native effectime.app UI:

1. **Wizard as a pop-up modal with the full settings set.** The purple "Intelligens beosztás varázsló" must open as a centered modal (like native), not slide in from the side, and expose the full settings (period start/end, optimization strategy, keep-existing / include-unmatched toggles, per-rule checklist) with a generate → preview → finalize flow — matching the attached screenshot (image 1).
2. **The calendar tab must show the full month timeline + filters** (image 2): one row per member, leave-type colour tints, approved/pending status dots, holidays/weekends, and the left filter sidebar (telephely, csapat, pozíció, szabadság típusa, státusz, képesség, helyszín).

## Key technical decisions

- **Reuse over re-implementation.** The embed iframe (`/#/embed/*`) mounts the full app inside all providers (`I18nProvider`, `ThemeProvider`, `QueryClientProvider`, `TooltipProvider`, Sonner). So embed views reuse native components (`CalendarFilterBar`, shadcn `Dialog`/`Select`/`Checkbox`, `@tanstack/react-virtual`) and the SAME i18n namespaces (`smart_batch.*`, `timeline_view.*`, `calendar_filter.*`) → automatic look-and-feel + localization parity across 8 locales.
- **Anonymous data boundary.** The guest is `anon` and can only read via the token RPC `get_embed_view_data` and write via `embed_assign_shift`. The native dialogs that `INSERT` directly or query tables cannot be reused as-is; embed twins were built that share the UI + pure ranking engine but swap the data boundary.
- **Additive RPC enrichment.** `get_embed_view_data` was extended (never renamed) so the same payload powers both the wizard's eligibility engine and the calendar's filters.

## Implementation details

- **Wizard:** self-fetches coverage data for the chosen `[start,end]` via `get_embed_view_data` (correct even when the range exceeds the visible grid), builds `MemberInput[]` / `EligibilityContext` / `sitePriorityMap` from the enriched payload, ranks with `evaluateEligibility` under the selected strategy (`role_match_first` / `priority_first` / `load_balanced`), previews matched/mismatch/member counts, then finalizes by looping `embed_assign_shift` (idempotent upsert on `workspace_id,user_id,shift_date`).
- **Calendar:** month grid virtualized by row; leave type → colour tint, status → dot colour; holidays/blocked → blue, weekend → zinc. Filters derived from payload (`offices`, `leave_types`, `skills`) + distinct member `team`/`business_role`/`city`. When the embed pins an `office`, the office filter row is hidden and members are hard-scoped to it.
- **RPC enrichment:** members `+team,+city,+weekly_capacity_hours,+base_working_hours,+skills[],+site_priorities[]`; `leave_requests +leave_type,+status` (approved+pending); new `leave_types[]`, `skills[]`.

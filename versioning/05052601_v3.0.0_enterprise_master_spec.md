## 1. Executive Context Summary

Effectime is a multi-tenant, multi-workspace enterprise SaaS platform for workforce operations, leave orchestration, capacity planning, approval workflows, and organizational intelligence. It is built on:

- **Frontend**: Vite + React + TypeScript, TanStack Router (`src/router.tsx`, `src/routeTree.gen.ts`), shadcn/ui + Tailwind, Capacitor wrapper for mobile builds.
- **Backend**: Supabase (Postgres + RLS + edge functions). Edge functions cover transactional email, Jira/ADO proxy, holiday sync, scheduled reports, account lifecycle, and instant-member provisioning.
- **Domain reality today**: dual-product monorepo (Syncfolk consumer calendar + PlannerMaster enterprise), schema-split as of 2026-04-29 multi-tenant migration, RLS via `has_enterprise_role()` / `is_enterprise_member()` SECURITY DEFINER functions.

This v3.0.0 directive does not rewrite the platform. It establishes the **structural framework** the existing modules must align to: Organization module as first-class; Onboarding and External Access Request matrices; Help system with contextual highlighting; Localization (EN + HU first); structured information architecture; non-regression contract.

## 3. Product Identity and Positioning

**Effectime is a workforce operations platform.** It is not a leave tool. The product narrative used in marketing, in-app copy, sales decks, and onboarding must consistently express the following positioning:

- **Primary value**: convert organizational time and capacity reality into reliable operational decisions (leave, staffing, onboarding, access, approvals, financial impact).
- **Differentiation**: a single integrated surface where calendar, capacity, approvals, organization, onboarding, access, and reporting reinforce each other instead of living in separate tools.
- **Audience**: enterprise HR ops, delivery managers, team leads, employees, and executives — each with role-specific landing surfaces.
- **Tone**: premium, calm, structured, power-user friendly. No decorative clutter.

## 6. Zero Functional Regression Contract

This contract is binding. All work derived from this directive must satisfy it.

### 6.1 Preservation classes

Every existing item is classified in one of four columns:

- **Preserve exactly** — no behavioral, visual, or API change permitted.
- **Visual redesign only** — pixels may change; behavior, contracts, permissions, and keyboard interactions must stay identical.
- **Enhance without changing behavior** — additive enrichment (new optional fields, new optional widgets) without altering current paths.
- **Risky change requiring explicit approval** — touching it requires a separate versioning artifact + business approval.

### 6.2 Default classifications

| Area | Class |
|---|---|
| Auth flows (`Auth.tsx`, OTP, Google, password reset, email activation) | Preserve exactly |
| Leave request lifecycle (submit, approve, reject, withdraw, substitute) | Preserve exactly |
| Approval chain semantics | Preserve exactly |
| Quotas & quota_transactions math (post v2.6.0 fix) | Preserve exactly |
| Conflict engine (`src/lib/conflictEngine.ts`) | Preserve exactly |
| Capacity engine (`src/lib/capacityEngine.ts`) | Preserve exactly |
| RLS policies & SECURITY DEFINER helpers | Preserve exactly |
| Email registry & transactional templates | Preserve exactly |
| Office coverage rule legacy/array fallback | Preserve exactly |
| Calendar filter system (`useCalendarFilterConfig`) | Preserve exactly |
| Audit log writes | Preserve exactly |
| Member profile sheet structure | Visual redesign only |
| Workspace dashboard layout | Visual redesign only |
| Header / global navigation | Visual redesign only (Help + Language additions) |
| Member creation form | Enhance without changing behavior (mandatory fields layered additively + soft-required during transition) |
| Position picker | Enhance without changing behavior (free-text path preserved, predefined path added alongside) |
| Onboarding | New (no regression risk on existing flows) |
| External Access matrix | New (no regression risk) |
| Help system | New (must not interfere with existing focus management or keyboard handlers) |
| Localization | New, but every UI string change must be regression-tested |

### 6.3 Hard rules

- No removal of any existing tab, page, route, button, filter, export, or notification without an explicit `risky-change` artifact and changelog `Removed` entry.
- No silent semantics drift on approval, leave conflict, capacity, or coverage logic.
- No RLS policy weakening. Any RLS change must be accompanied by a migration test and an audit-log impact note.
- No localization of database identifiers, enum codes, or stored permission keys — only display strings.
- No data migration that rewrites historical leave/approval/audit rows.

## 7. Non-Duplication and Rationalization Rules

The platform already contains many modules that overlap conceptually with the new requirements. Duplication is forbidden. The following rationalization rules apply.

### 7.1 People / Members / Organization

- The **Organization module** is the canonical owner of: hierarchy, leadership level, contractual relationship, industry, work categories, job families, employer-rights mapping.
- `MemberList`, `MemberProfileSheet`, `TeamManager`, `OfficeManager`, `BusinessRoleManager`, `RoleAllocationEditor` continue to exist but **read from and write to** the Organization module's canonical entities. They must not maintain a parallel hierarchy.
- A single membership record remains the source of truth per person per workspace. Org-unit, manager, contract type, and leadership level are fields on (or 1:1 extensions of) that membership, not a parallel record.

### 7.2 Positions and skills

- Position is one entity with two acquisition paths: free-text and catalog selection. The catalog is the new path; the free-text path remains valid.
- The role/category/skill catalog migration (`20260505110000_enterprise_role_skill_catalog.sql`) is the seed. The UI must use this catalog rather than introducing a second taxonomy.
- `SkillsManager` and skill expectations on positions share the same skill taxonomy; no duplicate skill list.

### 7.3 Approvals

- All approvals (leave, onboarding step, external access request) flow through a single approval-orchestration concept reusing `ApprovalChainManager` semantics. Onboarding and access do not introduce a parallel approval engine. They register policies into the existing engine.

### 7.4 Reporting

- Onboarding KPIs, access-completion KPIs, organization analytics, and command-center widgets must be implemented as datasets/views in the existing report builder. No second reporting subsystem.

### 7.5 Notifications

- All new flows route through `EnterpriseNotifications` + `NotificationPreferences` and the transactional email registry. No new notification bus.

### 7.6 Settings

- All workspace-scoped configuration lives under Workspace Settings. No new top-level settings node. Module-level settings (e.g., onboarding template editor) live within their module page, not Settings.

Top-level navigation is reorganized into seven primary destinations, in this fixed order:

1. **Home** — role-specific landing (employee / manager / ops / admin).
2. **Calendar** — Leave Calendar, Annual Grid, Timeline, Coverage Planner, Skill Capacity (existing tabs preserved).
3. **People** — Members, Teams, Offices, **Organization** (new sub-section: structure, leadership, contracts, industry, job families, org chart), Positions catalog, Skills.
4. **Workflows** — Approvals (inbox + chains), Substitutes, Onboarding, External Access Requests, Rule Templates.
5. **Resources** — Projects, Capacity, Utilization, Scenarios, Financials, Agile.
6. **Reports** — Library, Builder, Schedules, Pinned.
7. **Settings** — Workspace, Branding, Integrations, Notifications, Audit Log, Permissions, Localization, Holidays, Blocked dates, Company days, Imports/Exports.

### 8.1 Header anatomy (left to right on desktop)

`[Help (?)] [Logo] [Workspace switcher] [Primary nav] ··· [Search] [Language selector (flag)] [Notifications] [Profile menu]`

- **Help** is on the **left side of the header** with a question-mark icon — explicitly required by the directive.
- **Language selector** uses a circular flag and lives on the right cluster, just before notifications. Decision rationale (§9.2): keeping language and identity controls together matches enterprise convention (Atlassian, Workday, ServiceNow) and avoids visual competition with the Help control.

### 8.2 Mobile

- Bottom tab bar exposes Home, Calendar, Workflows, Reports, More. Help and Language are accessible from the top app bar (Help left, Language right).
- The Organization sub-section uses a sheet-style drill-down to keep org-chart usable on small screens.

### 8.3 Discoverability rules

- Every page exposes a stable breadcrumb: `Workspace › Section › Page`.
- Every page registers a `helpAnchorId` consumed by the Help system.
- Every page registers a `i18nNamespace` consumed by the localization system.
- No menu item appears unless the user has at least read permission on its content.

## 9. Header Strategy Including Help and Language Controls

### 9.1 Help control

- Icon: `?` (lucide `HelpCircle` or `CircleHelp`), 24×24 hit area minimum 40×40, left of the logo.
- Behavior on click: opens a **right-side drawer** (desktop ≥1024 px) or **full-screen sheet** (below 1024 px).
- Drawer contents:
  - **Header**: "Help — {currentSectionLocalized}" with the active section highlighted.
  - **Context chip row**: breadcrumb pills mirroring the user's current location with the active pill emphasized.
  - **Body**: contextual help content for the current `helpAnchorId` (overview, common tasks, keyboard shortcuts, related links, "open full manual").
  - **Footer**: search across the manual, language selector mirror, "report a problem" link.
- Highlighting: when opened, the page area corresponding to the `helpAnchorId` receives an outline ring (`ring-2 ring-primary/40`) and a soft pulse for ~1.2 s. The ring is removed on drawer close.
- Accessibility: drawer has `role="dialog"`, `aria-labelledby`, focus trap, ESC closes, return focus to Help button. Highlight ring is purely visual; assistive tech announces only the dialog.
- Localization: all help content keyed by `helpAnchorId` × language.

### 9.2 Language selector

- Trigger: circular flag of the active language. Country flags used: 🇬🇧 for English, 🇭🇺 for Hungarian. Accessible name announces the language (e.g., "Language: English").
- Click opens a popover dropdown listing all defined language sets (initially `en`, `hu`).
- Selection persists in:
  - `localStorage` (immediate frontend application),
  - the user's profile (`profiles.preferred_locale`) via Supabase update,
  - the auth JWT claim refresh path (next session),
  - a workspace-level **default** that admins can set, falling back to `en` when missing.
- Switching language must not reload the page. React Query caches stay; only translated strings re-render.
- Keyboard: `Tab`-reachable, `Enter`/`Space` opens, arrow-key navigation, `Esc` closes.
- Decision: language selector lives in the **header right cluster**, not above the menu bar. Rationale documented in §8.1.

## 10. Organization Module Specification

The Organization module is implemented as a new top-level destination under **People → Organization** with the following sub-pages:

- **Structure** — hierarchical tree of org units; create, rename, move, archive.
- **Leadership** — leadership-level catalog (Strategic, Operational, Technical, Execution) with assignment matrix to people.
- **Employer Rights** — definition of which roles exercise employer rights or delegated authority, with audit trail on every change.
- **Contracts** — catalog of contractual relationship types (employee, contractor, subcontractor, leased, consultant, temporary, custom), each editable, archivable, never deletable when in use.
- **Industry & Work Categories** — workspace's industry classification and the controlled vocabulary of work categories used elsewhere.
- **Job Families** — grouping of positions for analytics and onboarding template inheritance.
- **Org Chart** — auto-generated visual chart.

### 10.1 Mandatory metadata at member creation

The InviteMemberDialog and any programmatic member creation path must collect (with sensible defaults where appropriate):

- direct manager / supervisor (FK to membership; nullable only for the top of the tree, marked explicitly),
- organizational unit (FK),
- position (free-text or catalog FK),
- leadership level (enum, nullable for execution-only roles),
- contractual relationship type (FK to contract catalog),
- workspace assignment (already required),
- employer-rights flag (boolean, default false; only settable by admins),
- leadership category — strategic / operational / technical / execution / none.

Soft enforcement during transition: existing members without all fields are still valid, but a **completion banner** in MemberProfileSheet prompts admins to fill the gaps. After a configurable workspace cut-over date, fields become hard-required for new memberships.

### 10.2 Auto org-chart generation

- Source of truth: membership.manager_id graph + workspace/business-unit scope.
- Recompute trigger: on membership insert/update of `manager_id`, `org_unit_id`, `is_active`, or contract end. A debounced regeneration (≤30 s) writes to a snapshot table `enterprise_org_chart_snapshots`.
- Render: virtualized tree (D3 hierarchy + windowing) with collapse-by-default beyond depth 3 for orgs > 200 members.
- UX: focus mode (lock view to a sub-tree), search by name/role/skill, export to PNG/PDF, "open in profile" deep-link, mini-map for large trees.
- Cycle protection: insert/update trigger rejects cycles in `manager_id`.

### 10.3 Domain entities (suggested, additive)

```
enterprise_org_units            (id, workspace_id, parent_id, name, type, archived_at, created_at)
enterprise_leadership_levels    (id, workspace_id, code, label, sort_order)
enterprise_contract_types       (id, workspace_id, code, label, archived_at)
enterprise_industry             (id, workspace_id, code, label)
enterprise_work_categories      (id, workspace_id, code, label, parent_id)
enterprise_job_families         (id, workspace_id, code, label)
enterprise_org_chart_snapshots  (id, workspace_id, generated_at, payload jsonb)
-- additive columns on enterprise_memberships:
  org_unit_id, manager_id, leadership_level_id, contract_type_id,
  leadership_category text, employer_rights boolean default false
```

All new tables are workspace-scoped, RLS-protected via `is_enterprise_member()` / `has_enterprise_role()`.

## 11. Positions, Skills, and Seniority Specification

### 11.1 Acquisition paths

- **Free-text** (preserved): user types a position label, system stores it on the membership/allocation as today.
- **Catalog**: opens a drill-down picker:
  - Step 1: choose a category (work category / job family).
  - Step 2: choose a position within that category.
  - Step 3: review recommended skill expectations (junior/medior/senior layers).
  - Step 4: opt in or out per recommended skill before saving.

### 11.2 Recommended skill expectation inheritance

- Catalog positions carry default skill sets per seniority.
- Inheritance is **soft**: applying a position attaches the chosen subset to the membership/allocation record but does not lock future edits.
- Edits remain traceable: `enterprise_member_skills` rows store `source = 'template:<catalog_position_id>:<seniority>'` vs `'manual'`.
- Re-applying a template warns about overrides and offers merge options.

### 11.3 Seniority layers

- Seniority is a property of the membership-position record, not of the position catalog entry alone.
- Reporting and capacity matching use seniority as a weighting signal.

### 11.4 Domain entities (additive)

```
enterprise_position_catalog        (id, workspace_id, work_category_id, job_family_id, code, label, archived_at)
enterprise_position_skill_template (id, position_id, skill_id, seniority enum, required boolean)
-- additive on memberships / role allocations:
  position_catalog_id (nullable), seniority enum
```

Seed data is provided by the role-skill catalog migration already merged.

## 12. Onboarding and Access Request Matrix Specification

### 12.1 Onboarding domain

```
enterprise_onboarding_templates        (id, workspace_id, name, scope_org_unit_id?, scope_position_id?, version, status enum)
enterprise_onboarding_template_steps   (id, template_id, sort_order, type enum, owner_role, due_offset_days, mandatory bool, content_ref)
enterprise_onboarding_instances        (id, workspace_id, member_id, template_id, started_at, due_at, status enum)
enterprise_onboarding_step_completions (id, instance_id, step_id, status enum, completed_by, completed_at, evidence_ref)
enterprise_onboarding_acknowledgements (id, completion_id, ack_text_hash, ack_locale, ack_at)
```

Step types: `task`, `read`, `acknowledge`, `training`, `exam`, `approval`, `internal_permission`, `external_access`.

Escalation: each step has an optional `escalate_after_days` — overdue steps are surfaced in the Workflows inbox and trigger a notification via the existing email registry (new template `onboarding-escalation`).

### 12.2 Template instantiation

- Trigger: explicit "Start onboarding" action on a member, or auto-trigger on member creation when a template matches their (org_unit × position).
- Versioning: templates are immutable once published; new edits create a new version; existing instances reference their version.

### 12.3 External Access Request matrix

```
enterprise_access_systems            (id, workspace_id, name, kind enum [internal|external], default_owner_role, default_approver_role)
enterprise_access_templates          (id, workspace_id, scope_position_id?, scope_org_unit_id?, mandatory bool, sort_order)
enterprise_access_template_systems   (id, template_id, system_id, mandatory bool, optional bool, dependency_template_system_id?)
enterprise_access_requests           (id, workspace_id, member_id, system_id, template_id?, status enum, requested_at, decided_at, owner_id, approver_id)
enterprise_access_decisions          (id, request_id, action enum, actor_id, at, note)
```

Examples seeded: Jira, Confluence, Dynatrace, Outlook, ERP, Billing, Entry Control. Each is a row in `enterprise_access_systems` with workspace ownership; tenants can add custom systems.

Integration with Onboarding: an onboarding step of type `external_access` references an `enterprise_access_template_systems` row; completion of the access request flips the step to complete.

Approvals: routed through the existing approval chain engine. The matrix only declares **who must approve / who provides** — it does not implement a parallel approval state machine.

### 12.4 UX outline

- **Onboarding inbox** — per-member checklist with progress bar, owner column, deadline column, blocking dependencies.
- **Template library** — workspace + org-unit + position scoped templates with version history.
- **Access matrix** — pivot-style grid: positions × systems, cell shows `required / optional / not applicable`. Editable by admins.

## 14. UX/UI System and Interaction Design Guidance

- **Design tokens**: Tailwind theme + CSS variables already in `index.css` and `tailwind.config.ts`. Continue using them; no new color system.
- **Spacing**: 4 / 8 / 12 / 16 / 24 / 32 scale only.
- **Density**: Default comfortable; "compact" mode for power users via Workspace Settings.
- **Hierarchy**: Page title (28/32 semibold), section title (20/28 semibold), label (12/16 medium uppercase tracking-wide), body (14/20).
- **Components**: shadcn/ui primitives are canonical; do not re-skin. Compose, don't fork.
- **Tables**: shared `DataTable` pattern with column visibility memory persisted via `useWorkspaceSectionState`.
- **Drawers vs modals**: drawers for create/edit flows that need context behind them; modals for confirmation only.
- **Empty / loading / error / success states**: every list and chart implements all four with copy keyed by i18n.
- **Mobile**: every primary action reachable in ≤2 taps from bottom tab bar.
- **Keyboard**: `?` opens Help, `g h` Home, `g c` Calendar, `g p` People, `/` focus search, `j/k` row navigation in tables, `Enter` open, `e` edit, `Esc` close.
- **Accessibility target**: WCAG 2.2 AA. Color contrast ≥ 4.5:1 for text, focus ring visible on every interactive element, semantic headings.

- **Data layer**: continue Supabase client + React Query (existing pattern). New queries follow `['enterprise', '<domain>', ...keys]` namespace.
- **i18n**: introduce `i18next` with namespaces per module (`common`, `calendar`, `leave`, `approvals`, `people`, `organization`, `onboarding`, `access`, `reports`, `settings`, `help`). Lazy-load Hungarian by default for Hungarian users; English bundled.
- **Help registry**: `src/lib/help/registry.ts` exposes `registerHelpAnchor(id, contentResolver)` and `useHelpAnchor(id)`. Pages call `useHelpAnchor` on mount.
- **Permissions**: continue `useEnterprisePermissions`. New permissions added: `org_unit:manage`, `contract_types:manage`, `position_catalog:manage`, `onboarding_templates:manage`, `onboarding_instances:manage`, `access_systems:manage`, `access_requests:approve`, `localization:manage`.
- **State**: avoid global stores beyond what already exists. Help drawer state and language selection live in their own contexts.
- **Performance**: virtualize tables > 200 rows, virtualize org-chart, code-split per top-level route.

## 16. Backend and Domain Architecture Guidance

- **Bounded contexts**: Identity & Auth, Organization, Membership, Time & Calendar, Leave & Approvals, Onboarding, Access, Capacity & Resources, Integrations, Reporting, Notifications, Audit & Compliance.
- **Schema**: continue per-product schema split. New tables under `plannermaster` (or current enterprise namespace), additive columns only.
- **Migrations**: every change goes through `supabase/migrations` with a date-stamped filename. No direct production schema edits.
- **RLS**: extend `is_enterprise_member()` and `has_enterprise_role()` patterns. New permissions slot into the existing role permission table.
- **Triggers**:
  - Cycle-prevention on `enterprise_memberships.manager_id` updates.
  - Org-chart snapshot enqueue trigger (NOTIFY-based or pg_cron throttled).
  - Onboarding step auto-completion when external access request is approved.
- **Edge functions**: reuse existing pattern. New optional functions:
  - `org-chart-rebuild` — scheduled regen for large workspaces.
  - `onboarding-escalator` — pg_cron daily, fires escalations.
- **Audit**: every Organization, Onboarding, Access, and permission change writes to the existing audit log with `entity_type`, `entity_id`, `action`, `actor_id`, `before_after_diff`.
- **Idempotency**: new mutation endpoints accept `Idempotency-Key` headers (frontend generates per-action UUIDs).

## 17. Data Model and Integrity Considerations

- All new FKs use `ON DELETE RESTRICT` for catalog entries that are referenced from instance data; `ON DELETE CASCADE` only inside a single bounded context (e.g., template → template_steps).
- Every workspace-scoped table includes `workspace_id` and an RLS policy validating membership.
- Every catalog row exposes `archived_at` rather than hard-deleting; UI hides archived by default.
- Every new enum is a Postgres `text` column with a CHECK constraint, never a real enum type (per lessons learnt — easier evolution).
- Date/time columns are `timestamptz`, never `timestamp`.
- Money columns (financials) use `numeric(14,2)` with explicit currency code column.
- Indexes added on (`workspace_id`, foreign keys used by lists) per usage.
- Soft-required member fields backfilled to `NULL` and surfaced as completion gaps rather than blocking existing data.

## 18. Localization Architecture (English + Hungarian)

### 18.1 Frontend

- Library: `i18next` + `react-i18next`. JSON resource bundles under `src/i18n/<locale>/<namespace>.json`.
- Default locale: `en`. Workspace can configure default; user override wins.
- Detection order: explicit user setting → workspace default → browser → `en`.
- Missing-key behavior: development logs a warning, production renders the key wrapped in a marker (`⟦key⟧`) only in non-prod environments; in prod renders the English fallback.
- Number, date, currency: `Intl.*` with workspace timezone + locale.
- Pluralization: i18next plural rules (Hungarian uses `one`/`other`).

### 18.2 Backend

- `profiles.preferred_locale text` column (additive migration).
- `enterprise_workspaces.default_locale text` column.
- Edge functions sending email read recipient locale and select template variant. Email templates already in `_shared/transactional-email-templates` are extended with `_hu.tsx` variants where text exists.
- Stored content (notification subjects, help content, onboarding step titles, manual chunks) lives in localized tables: `<entity>_translations (entity_id, locale, field, value)`.

### 18.3 Translation key strategy

- Keys are dotted, lowercased, English: `people.organization.actions.add_unit`.
- Never localize keys themselves.
- Reusable strings live in `common`; never duplicate.

### 18.4 Fallbacks

- Locale fallback chain: `hu` → `en`.
- Template fallback chain: localized → English.

### 18.5 Admin manageability

- Settings → Localization page lists languages, allows enable/disable, sets workspace default, shows missing-key counts per namespace, supports CSV export/import for translator workflows.
- Adding a third language is a configuration change + bundle drop, not a code change.

## 19. Help System Specification

- **Content model**: each `helpAnchorId` has a markdown chunk per locale, plus structured metadata: `title`, `summary`, `commonTasks[]`, `keyboardShortcuts[]`, `relatedAnchors[]`, `manualLink`.
- **Storage**: bundled with the app for v1 (`src/i18n/<locale>/help/<anchor>.md`). v2 promotes admin-editable workspace-specific overrides to a database table `enterprise_help_overrides`.
- **Registry**: pages call `useHelpAnchor('people.organization.structure')` on mount; the Help drawer reads the current anchor from context.
- **Highlighting**: pages mark highlightable regions with `data-help-region="<anchor>"`. The drawer adds the ring class to the matching element.
- **Search**: client-side full-text over loaded help bundles, language-aware.
- **Telemetry**: anonymized counters of which anchors are opened, to inform manual updates (privacy-safe, k≥5 aggregation).
- **Failure mode**: if no anchor is registered, drawer shows the section's overview content based on the current breadcrumb.

## 21. Accessibility and Keyboard Support Requirements

- Standard: WCAG 2.2 AA across all new and visually redesigned surfaces.
- Color: contrast ≥ 4.5:1 for text, ≥ 3:1 for non-text. No information conveyed by color alone (badges include icon + label).
- Focus: visible focus ring on every interactive element; no `outline: none` overrides.
- Keyboard: shortcuts listed in §14; all dialogs/drawers focus-trapped with ESC dismissal.
- Screen readers: every icon-only button has `aria-label`; live regions announce async successes/failures; tables expose proper header semantics.
- Motion: respect `prefers-reduced-motion`; help highlight pulse disabled when set.
- Internationalization affects accessibility: `lang` attribute on `<html>` updates with locale; date/number rendering matches locale.

# v3.51.3 — Release identity and deployment attestation

Date: 2026-07-18
Status: implemented on `codex/release-identity-attestation`; production publish
and Edge rollout are not part of this source change.

## Objective

Make a web/Edge deployment attributable to one immutable, full Git commit SHA
and fail visibly when the deployed layers are unknown or different. This closes
the source-side provenance gap found during the enterprise audit; it does not
override database, staging, backup/restore or provider-authorization NO-GO gates.

## Contracts

- Build input: `EFFECTIME_RELEASE_SHA`, then `GITHUB_SHA`; local non-CI builds may
  resolve Git HEAD. Values must be exactly 40 hexadecimal characters. CI refuses
  missing identity and conflicting inputs.
- Web artifact: `/.well-known/effectime-release.json`, deterministic and free of
  timestamps. It contains a bounded per-file SHA-256 inventory; the same SHA and
  reviewed Edge source digest are compiled into the client. Dirty builds are
  explicit and cannot be release-attested.
- Edge runtime: `EFFECTIME_RELEASE_SHA`. Valid values produce
  `{ sha, sourceTreeSha256, status: "identified" }`,
  `X-Effectime-Release-SHA` and `X-Effectime-Edge-Source-SHA256`; invalid/missing
  values keep the immutable source hash but produce `status: "unknown"` without
  the release-SHA header.
- Public API health exposes the identity as `data.release`; superadmin
  `platform-version` exposes top-level `release`.
- Superadmin displays web/Edge match, mismatch, unknown and retryable error.
- `release:verify:deployment` rejects a missing/malformed/dirty web manifest,
  wrong provider deployment ID, file byte/hash drift, unknown/mismatched Edge
  body or either Edge header, and any value different from the expected Git SHA,
  web fingerprint or Edge source digest. The API key is accepted only from
  `EFFECTIME_PUBLIC_API_KEY` and sent only to the exact HTTPS
  `EFFECTIME_EDGE_ORIGIN`.

## Compatibility and security

- Web path, Edge JSON fields and response header are additive.
- Missing Edge configuration remains visible but does not crash unrelated API
  routes; it cannot be presented as a successful match.
- No secret is embedded in the client, command arguments, log messages or
  manifest. A scoped Effectime API key is needed only for authenticated live
  Edge verification.
- The embedded source-tree digest prevents a stale Edge bundle from passing by
  changing only the runtime SHA label. Each independently deployed function must
  still be mapped to its provider deployment/version ID; one health endpoint
  cannot prove the provider version of the other functions.

## Rollback

Revert this unit and rebuild/redeploy the previous immutable artifact. Do not
remove or falsify a release identity on an already-running artifact. A production
rollback must restore matching web and Edge layers and record their provider IDs.

# v3.18.0 — Gamification & engagement layer (Top-20 Rank 14)

**Date:** 2026-05-13
**Branch:** `claude/review-project-requirements-lR0Wp`
**Strategy doc:** Effectime Enterprise — Top 20 Value-Rocket Growth Strategy

## Why this is the first delivery against the strategy

The user supplied a 20-feature growth strategy with implementation prompts
and value estimates. A deep audit (`db-audit/feature_gap_audit.md`)
classified the 20 ranks as: 5 DONE, 6 PARTIAL, 9 MISSING. Shipping all
9 missing in a single commit is unrealistic — full-stack delivery with
DB + edge fn + UI + 5-locale i18n + route + tier + governance per
feature would span weeks.

**Rank 14 (Gamification)** was selected as v3.18.0 because:
- It is one of the 6 completely green-field ranks (no catalog entry, no
  table, no UI). Building it from scratch demonstrates the canonical
  full-stack delivery template.
- It is purely additive — no existing flow is touched, so regression
  risk is near-zero.
- It is self-contained — no upstream dependency on Rank 9 (API platform)
  or Rank 7 (PWA).
- It demonstrates compliance with the v3.7.2 localization controller
  (5 locales, 29 keys each), the v3.16.0 routing rule (`/w/:workspaceId`),
  the v3.17.0 tier governance (mapped to Pro + Enterprise, not Freemium),
  and the v3.15.4 audit guarantee (the new RPC is the ONLY writer of the
  achievement tables — direct INSERT is policy-blocked).

## Implementation summary

### DB

| Object | Type | Purpose |
|---|---|---|
| `engagement_achievements` | table | System-wide catalog of achievement definitions |
| `engagement_member_achievements` | table | Per-member earned badges (immutable history) |
| `engagement_streaks` | table | Per-member streak counters |
| `enterprise_memberships.gamification_opt_out` | column | Per-member opt-out preference |
| `enterprise_workspaces.gamification_enabled` | column | Per-workspace toggle |
| `engagement_record_event(uuid, uuid, text)` | function | SECURITY DEFINER award helper — the ONLY writer |

7 seed achievements: punctual_5_days, punctual_30_days, punctual_100_days,
planner_3_advance, coverage_hero_1, coverage_hero_10, profile_complete.

### RLS contracts

- `engagement_achievements`: read = authenticated; write = platform admin only.
- `engagement_member_achievements`: read = own member OR workspace owner/resourceAssistant; **direct INSERT blocked** (only the SECURITY DEFINER RPC writes).
- `engagement_streaks`: read = same as above; **direct INSERT blocked**.

### Award path

Front-end calls (e.g. clock-in success):
```ts
await recordEngagementEvent(workspaceId, membershipId, 'clock_in_on_time');
```

Server flow:
1. `auth.uid()` non-null → caller is the member or owner/RA.
2. Workspace `gamification_enabled` check → bail if disabled.
3. Member `gamification_opt_out` check → bail if opted out.
4. Authorization check (caller is member OR owner/RA).
5. Upsert streak counter for the mapped `streak_type`.
6. For each achievement whose `trigger_event` matches AND `threshold` ≤ new streak:
   - If `is_repeatable` OR not yet earned, INSERT into earned table.
7. Return `{ ok, streak, awarded[] }`.

### Frontend

- `useEngagement.ts` (hook + helpers).
- `AchievementsPanel.tsx` (read-only badge wall + streak counters).
- Wired into `EmployeeDashboard` (self-service area). The panel renders
  for any member who has a membership in the workspace and reads ARE
  RLS-scoped: members see only their own; owners/RAs see team members'.

### Feature catalog + tier mapping

3 new `feature_keys`:

| feature_key | route_path | menu_path | tiers |
|---|---|---|---|
| `gamification_dashboard` | `/w/:workspaceId` | `Profile › Achievements` | Pro, Enterprise |
| `gamification_badges` | `/w/:workspaceId` | `Profile › Achievements › Badges` | Pro, Enterprise |
| `gamification_streaks` | `/w/:workspaceId` | `Profile › Achievements › Streaks` | Pro, Enterprise |

Dependency graph: `gamification_badges` and `gamification_streaks` both
depend on `gamification_dashboard`.

**Freemium intentionally excluded** — engagement features drive paying-customer
retention and would dilute the upsell rationale if free.

### Localization

29 keys × 5 locales (en, hu, cs, sk, pl) = 145 strings total. Inserted at
the END of each resource file via a single Python pass (atomic, no
half-translated state).

1. As a workspace member, open the self-service dashboard (`/w/<id>?tab=my-portal`).
2. Verify the AchievementsPanel renders below the regular widgets.
3. Verify all 7 system achievements appear in the badge wall, all locked initially.
4. Verify streak counters section says "no streaks yet" with the i18n-localized hint.
5. As a workspace owner, run via SQL editor:
   ```sql
   SELECT engagement_record_event('<workspace_id>', '<your_membership_id>', 'profile_completed');
   ```
   Expect `awarded` to include `profile_complete`. Refresh dashboard → "Complete profile" badge is now amber/earned.
6. Verify Czech/Slovak/Polish UI shows localized achievement names (switch locale in user profile).

## Out of scope (deferred to follow-up)

- **Auto-trigger of `engagement_record_event` from existing UI flows** —
  the helper is called nowhere yet. Wiring it into clock-in success,
  leave-submit-with-2-week-notice, shift-swap-accept, and profile-complete
  events is the next polish PR. Currently events must be triggered
  manually via the RPC for testing.
- **Manager team view** — a "team engagement heatmap" was in the strategy
  doc but is deferred; the current panel is per-member.
- **Anonymous benchmarks** ("you're in the top 25%") — requires
  aggregation queries and consent UX; deferred.
- **Push notification on achievement earn** — depends on Rank 7 (PWA/FCM).

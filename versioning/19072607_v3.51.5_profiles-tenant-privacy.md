## Goal and evidence

`public.profiles` is a global identity table. Its historical permissive policy
allowed any authenticated user to read complete non-token profile rows, and the
birthday/anniversary widget issued an unscoped
`profiles.select('user_id,display_name,preferences')` query before filtering the
already downloaded data by workspace in the browser. The `preferences` JSON can
contain birthdays and activation material, so client-side filtering was not a
tenant or confidentiality boundary.

The repair must retain the canonical shared profile source used by web, Android
and iOS while enforcing tenant visibility and column confidentiality on the
server. The milestone UI must receive only the data it needs, through one
versioned, permission- and entitlement-checked contract.

## Implemented database boundary

- Adds an active-membership lookup index on `(user_id, workspace_id)` without
  replacing the existing workspace/user uniqueness contract.
- Adds a tenant-safe permissive profile visibility basis and a restrictive
  guard. An authenticated caller may see their own profile, platform admins may
  retain directory resolution, and an active workspace member may resolve
  profiles connected to the same workspace. A suspended caller receives only
  self visibility; another tenant is not visible.
- Keeps historical/suspended target profiles resolvable to an active member of
  the same workspace for existing audit/history UI. The milestone RPC is
  narrower and returns active targets only.
- Treats same-workspace name/avatar resolution as an intentional organizational
  identity contract, independent of members-administration permission. It exposes
  only `user_id`, `display_name` and `avatar_url`; the milestone path performs
  its own stricter permission and entitlement checks.
- Revokes browser table privileges and dynamically revokes every profile-column
  ACL found in the PostgreSQL catalog, including unknown drift-only columns,
  before re-granting authenticated `SELECT` only on `user_id`, `display_name` and
  `avatar_url`. Existing service-role table/column access is preserved.
- Limits authenticated self-service `UPDATE` to `display_name`, `avatar_url` and
  `preferred_locale`. Matching permissive and restrictive own-row update policies
  prevent an unknown additional permissive policy from widening the target row.
  Browser insert, delete, preference updates and temporary/activation material
  remain denied.
- Replaces the two-locale database check with one validated constraint matching
  the eight runtime locales: `en`, `hu`, `cs`, `sk`, `pl`, `de`, `at`, `ro`.
- Adds a private, runtime-inaccessible birthday parser that fails safely for
  missing or malformed dates.
- Adds authenticated-only `get_my_profile_locale_v1()` for the current
  `auth.uid()`; locale is not a coworker directory attribute and cannot be read
  directly through the browser table projection.
- Adds authenticated-only `update_my_workspace_profile_display_name_v1(
  p_workspace_id uuid, p_membership_id uuid, p_display_name text)`. It validates
  and trims the name, requires an active
  membership belonging to `auth.uid()` in the supplied workspace, and therefore
  cannot let a tenant administrator change another user's global identity.
  Missing profiles fail closed; auth lifecycle remains the only profile creator.
- Adds `get_workspace_member_milestones_v1(p_workspace_id uuid)` as a stable
  `SECURITY DEFINER` function with `search_path=pg_catalog`. It requires an
  active workspace caller, `calendar:readonly`, `members:readonly`, and both
  `birthday_widget` and `members_list` entitlements.
- Grants all three public RPCs only to `authenticated`. `anon`, `service_role`
  and `PUBLIC` cannot execute them, and runtime roles cannot execute the private
  parser.
- Returns exactly five milestone fields: membership ID, display name, milestone
  type, month and day. It never returns user ID, birth year, raw preferences,
  email, activation state or temporary-access material.

## Implemented shared client contract

- Replaces the widget's global profile/preferences query with one exact call to
  `get_workspace_member_milestones_v1`. There is no raw-profile fallback.
- Routes `MemberProfileSheet` display-name changes through the bounded
  `workspaceMemberProfileApi` adapter and self-only profile RPC instead of a
  direct profile upsert. The field is disabled for a coworker's global identity;
  UUID/name/response validation and a 15-second timeout fail closed with bounded
  non-PII errors.
- Uses the same React/TypeScript adapter and Supabase client for web, Android
  and iOS; no mobile-only database, privileged key or alternative business API
  is introduced.
- Validates the workspace UUID before dispatch, bounds the request to 15
  seconds, supports caller cancellation and maps failures to bounded,
  non-sensitive error codes.
- Rejects malformed or over-broad responses, invalid UUIDs, dates, milestone
  types and duplicate membership/type rows. A member may legitimately have one
  birthday and one work anniversary.
- Clears old workspace data immediately and uses abort/generation guards so a
  late response cannot overwrite a newer tenant's loading, error or data state.
- Provides explicit loading, retryable error and ready states. Loading/error
  announcements, native button semantics, focus behavior, semantic list/time
  markup and decorative-icon treatment are included.
- Uses the canonical workspace timezone in the server anniversary calculation
  and client calendar. Focus/visibility resume and the minute clock prevent a
  suspended mobile shell from retaining yesterday's result; year-boundary/DST
  sorting and February 29 remain exact.
- Mounts the widget only after resolved `members` visibility and the required
  `birthday_widget` plus `members_list` feature state.
- Hydrates own locale through `get_my_profile_locale_v1`, persists changes to
  the own row by `profiles.user_id`, serializes writes, and rejects stale auth or
  slower persistence responses. Workspace translation overrides clear immediately
  on tenant switch and use a separate generation guard.
- Adds a TypeScript-AST contract across reachable browser source. Literal profile
  reads are limited to the three safe directory columns, updates to the three
  self-service columns, and insert/upsert/delete is forbidden. The unreferenced
  legacy `CsvImportPanel` remains explicitly quarantined and must stay unreachable.
- Changes `CapacityFit` to accept an assignee only when its normalized name has
  exactly one `display_name` match in the safe workspace directory. Duplicate,
  empty and email-only identities stay in the localized unmatched row, planned
  hours are counted once, and browser profile email is neither selected nor shown.
- Introduces no new runtime dependency.

## Compatibility and rollout

The three new RPCs are additive. Existing directory consumers that select only
`user_id`, `display_name` and `avatar_url` retain their server-authorized path;
consumers selecting `id` or `preferred_locale` must migrate, with own-locale
hydration using `get_my_profile_locale_v1`. The table
privilege tightening is an intentional security boundary: a previously deployed
birthday widget or other client that directly selects `profiles.preferences`
will fail closed after database apply and must not regain that access as a
compatibility fallback.

Required rollout after migration-history reconciliation:

1. verify backup/PITR and restore production-like data in isolation;
2. reconcile the target migration history and review the exact policy, grant,
   function and constraint diff;
3. inventory all live direct profile consumers and prove through the AST caller
   contract that supported directory paths use only the safe projection;
4. apply the database migration first, refresh PostgREST schema cache, and run
   two-tenant RLS/ACL plus exact locale/self-profile/milestone RPC acceptance;
5. deploy the new web artifact immediately after the server boundary, then use
   the same attestable source for Android/iOS candidates;
6. verify authenticated workspace switching, permission/feature downgrade,
   retry behavior and the live release identity.

DB-first closes the exposure before the new UI is trusted, but it is not fully
functionally backward compatible. Use one controlled maintenance window: prove
old-client fail-closed and new-client success on restored staging, apply the DB
and refresh PostgREST schema cache, then release the attestable client
immediately. During the shortest possible window an old birthday widget cannot
load milestones; it must fail closed rather than fall back to global preferences.

The three new public RPCs have not been added manually to the generated Supabase
type file. The adapters isolate exact narrow interfaces while live schema
provenance and generated types are reconciled. Regeneration from an unverified
drifting schema would not be valid release evidence.

## Rollback and recovery

Before database apply, the candidate can be reverted normally in git. After
apply, do not restore the global profile/preferences grants or remove the
restrictive tenant guard to recover UI behavior. A client rollback must use a
compatible fail-closed state; it must not restore the former raw-profile query.

If the new server contract needs correction, keep the privacy boundary in place
and ship a separately reviewed forward migration. Any data-integrity issue
requires the verified backup/restore procedure. The private parser and
versioned RPCs can remain installed while a corrected client is rolled out.

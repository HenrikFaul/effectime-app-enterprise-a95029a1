## 1. Demo workspace flow

### 1.1 Backend
- **New edge function** `supabase/functions/seed-demo-workspace/index.ts`:
  - Authorizes via the calling user's JWT.
  - Calls `create_workspace_with_owner` RPC (SECURITY DEFINER, requires `auth.uid()`).
  - Creates 7 real `auth.users` via `admin.auth.admin.createUser({ email_confirm: true, app_metadata: { is_demo_persona: true } })`. This avoids any schema changes and lets every existing component (profile lookups, leave_requests, allocations, member_skills) work transparently.
  - Seeds the workspace across 13 modules (offices, teams, leave types, holidays, skills, memberships, member-skills, allocations, leave_requests, daily rules, office coverage rules, allowances, audit log).
  - Stores the demo persona user_ids in `enterprise_workspaces.settings.demo_user_ids` for later cleanup.
  - Returns a summary `{ members, offices, teams, leave_types, skills, leave_requests, holidays }` consumed by the UI toast.

- **New edge function** `supabase/functions/cleanup-demo-workspace/index.ts`:
  - Owner-only deletion path.
  - Deletes the workspace (cascades all workspace-scoped tables via existing FK).
  - Deletes the demo `auth.users` (cascades `profiles` via FK), preventing orphans.

### 1.2 Frontend
- **`CreateWorkspaceDialog`** redesigned: existing manual creation flow preserved; new accent-card panel below the description field offers `Demo munkaterület létrehozása`. The button calls `seed-demo-workspace`, displays a loading state, and reports the seeded counts in a success toast.

### 1.3 Realism guardrails
The demo dataset is intentionally varied so that every module can be tested:
- Leave statuses cover approved (past + future), pending, rejected, sick leave.
- Members are spread across 3 cities + 4 teams + 4 different positions.
- Allocations + skills + holidays + coverage rules give the capacity engine real signal.

## 2. Position catalog wiring

### 2.1 Resources → Pozíciók (BusinessRoleManager)
Previously the manager only had a free-text input ("Új pozíció neve…"). Users were typing positions blindly even though the global catalog already contained hundreds of pre-seeded roles.

**Fix**: a `Katalógus megnyitása` button next to the free-text input launches the existing `PositionPickerDialog`. Picking a position from the catalog appends it as a new role group, ready for member allocation. The free-text path is preserved for ad-hoc positions.

### 2.2 PositionPickerDialog — global catalog fallback
The dialog used to read only from `enterprise_workspace_role_categories` and `enterprise_workspace_roles` (the workspace customization layer). For fresh workspaces, those tables are empty, so the dialog showed a useless empty state — even though the global `enterprise_catalog_*` tables had 558+ skills.

**Fix**: when the workspace tables return zero rows, fall back to the global catalog (`enterprise_catalog_categories` / `enterprise_catalog_roles`). When the user reviews skills, the global `enterprise_catalog_role_skills` + `enterprise_catalog_skills` are read instead. A small banner ("Globális katalógusból olvasunk…") makes the source visible. When the workspace customization layer has rows, behavior is unchanged.

## 3. Jira sync 500 error

### 3.1 Root cause
`jiraSyncProjectConfig` in `jira-devops-proxy` called:
```
GET /rest/api/3/issuetype/project?projectId={projectKey}
```
The `projectId` query parameter expects a **numeric project ID** (e.g., `10000`), not a project key (e.g., `SYN`). Jira returns 500 when the value is non-numeric.

### 3.2 Fix
Replaced with `GET /rest/api/3/project/{projectIdOrKey}` which returns the project including its `issueTypes` array. Falls back to `GET /rest/api/3/issue/createmeta?projectKeys={key}&expand=projects.issuetypes` for older Jira tenants. Label/component discovery is now best-effort and only warns on failure rather than aborting the whole sync.

## 4. In-app Jira issue editor

### 4.1 Backend (jira-devops-proxy new actions)
- `get_issue` — fetches a single Jira issue with all fields + transitions in parallel.
- `get_transitions` — lists allowed status transitions for the issue.
- `search_assignable_users` — type-ahead user search via `/rest/api/3/user/assignable/search`.
- `update_issue` extended to accept: `description`, `priority`, `due_date`, `story_points` (customfield_10016), `assignee_account_id` (null clears), `status_transition_id` (routed through the transitions endpoint, not the field update endpoint). After successful save, the proxy re-reads the issue and upserts the fresh row into `enterprise_agile_issues` so the local cache reflects the update without a separate refresh.

### 4.2 Frontend (`JiraIssueEditor`)
- Dialog component that loads a Jira ticket and renders editable fields:
  - Free text: summary, description, labels (comma-separated), due date.
  - Numeric: story points.
  - Select: priority, status (via available transitions), assignee (with type-ahead).
- Read-only metadata footer: reporter, sprint, parent, components, created/updated timestamps.
- Direct link to the original Jira ticket is preserved.
- "Mentés a Jira-ba" sends only the diff. After save, reloads the issue to show the post-transition state.

### 4.3 Wiring
- **BacklogBrowser**: clicking the ticket summary or the new pencil icon opens the editor.
- **AgileBoards** (Kanban / Scrum / Gantt): cards and rows are now keyboard- and click-actionable. The external Jira link still navigates externally (`stopPropagation`).

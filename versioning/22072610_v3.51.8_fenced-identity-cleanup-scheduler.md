## Problem and evidence

The v3.51.7 compensation saga was durable, but its queue claim had no fencing
token and there was no reviewed recurring single-flight scheduler. A delayed
worker could therefore not prove that it still owned a replaced lease. The
legacy temporary-profile cleanup also mixed a separate data-retention concern
with the identity saga.

## Implemented boundary

- Every claimed created-identity job receives an opaque lease token. Prepare,
  complete and fail require the exact unexpired token; expiry is immutable and a
  late worker cannot mutate a replacement lease.
- A private singleton records the active run, redacted overlap count, bounded
  completion counters and safe error code. The Edge worker processes at most ten
  jobs, uses a 150-second singleton lease, a 110-second total deadline and
  15-second network-operation limits.
- `cleanup-created-identities` is a dedicated POST-only worker. The scheduled
  route requires the exact target-project legacy anon JWT in both gateway
  headers, an independent high-entropy trigger secret and an exact versioned
  body. The manual route is exact service-role only.
- The owner-only installer validates the exact 20-character project ref and one
  value for each required Vault name. Its five-minute command resolves URL and
  credentials at runtime and persists no literal secret. The migration itself
  creates no cron job.
- `cleanup-temp-users` remains a separate legacy temporary-profile responsibility
  and cannot claim or reconcile the created-identity saga. A service-only
  eligible-row claim, 120-second per-user token and profile/event mutation guards
  freeze the deletion decision across Auth. The exact lease is renewed after
  Auth lookup and immediately before each delete; exact absence plus an
  unexpired token is required before one atomic DB completion. Expired orphan
  tokens rotate through a new claim, and the exact same-owner Auth cascade is
  admitted only for a sole linked profile. The Edge path has abortable
  ten-second operations and a 90-second total deadline.
- The backend contract is shared by web, Android and iOS; no mobile-only data
  store, cleanup behavior or privileged key is introduced.

## Rollout and NO-GO boundary

Required order: verified backup/history reconcile → restored-staging database
migration → the four matching Edge artifacts from one frozen SHA → manual worker
smoke → Vault validation → scheduler installation last. Retain the pg_net HTTP
response and correlate it with singleton worker state; cron enqueue success or
one HTTP 200 is not sufficient evidence. Pause/unschedule is the first rollback
action. Production remains **NO-GO** while migration drift, 131/131 clean replay,
restored-staging fault injection, provider deployment IDs and live release
attestation are unresolved.

# v3.42.0 — DB: canonicalize `features.route_path` to `/w/:workspaceId/...`

**Date:** 2026-05-17  
**Type:** schema / data migration (UPDATE only, no DROP)  
**Risk:** very low — `features.route_path` is consumed only by the
Superadmin Feature Tiers tree for grouping and search. No live browser
navigation depends on it.

## What changed

Two defects in `public.features.route_path` resolved in one transactional
migration:

| Defect | Rows | Before | After |
|---|---:|---|---|
| Legacy `/app/...` prefix (pre-v3.16.0) | 111 | `/app/resources/agile/gantt` | `/w/:workspaceId/resources/agile/gantt` |
| Truncated workspace path (no leaf) | 31 | `/w/:workspaceId` | `/w/:workspaceId/<derived-from-menu_path>` |

Post-migration distribution: **142** canonical workspace-scoped, **30**
platform-level, **0** legacy, **0** truncated.

The 31 derived leaf paths come from each row's existing `menu_path`
breadcrumb (lowercase, non-alphanumeric runs collapsed to a single `-`).
No structure was invented — `menu_path` already encoded the intended
location.

## Why this version

Aligns with `.governance/ui_ux_rules.md` § "Workspace identifier in URL"
(non-negotiable from v3.16.0): every workspace-scoped path uses the
`/w/<workspaceId>/<rest>` shape.

## Anti-regression measures

The migration carries **five hard invariants** that abort the transaction
on any unexpected outcome:

- **A:** 0 rows left with `/app/` prefix.
- **B:** 0 rows left with bare `/w/:workspaceId`.
- **C:** every value still passes the Superadmin validation regex
  `^/[A-Za-z0-9/_\-:$.{}-]*$` consumed by `FeatureTiersTab.tsx`.
- **D:** same non-null `route_path` row count pre/post.
- **E:** every rewritten `/app` row now lands under `/w/:workspaceId/`.

All five passed.

## Affected UX surfaces

Only one: the Superadmin → Feature Tiers tab. The tree now groups under
the new canonical labels. No end-user URL changed; the React Router shell
(`src/App.tsx`) does not match these strings as browser routes.

## What this PR does NOT do

Per the no-regression mandate, none of the following were touched:

- The `route_path` column itself (no rename, no drop, no FK).
- Any `route_path` consumer code.
- The pre-existing `(route_path, menu_path)` duplicate pairs in the
  Clock-in family and Shift Marketplace family. These are pre-existing
  data-shape questions for product, surfaced by the canonicalisation
  but not introduced by it.
- The role / permission / business_role domain — documented in
  `role_model_normalization.md`, blocked on three product answers.
- The `enterprise_role_permissions.feature_key` namespace collision —
  documented in `terminology_audit.md`, scheduled for a separate PR
  after stakeholder review.

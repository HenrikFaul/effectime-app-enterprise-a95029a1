## Compatibility notes

### Direct workspace user creation

`create-workspace-user` now returns HTTP 409 with
`DIRECT_CREATE_DISABLED`. The previous tenant-admin flow created a globally
confirmed identity without mailbox proof and allowed an email address to be
pre-claimed before its owner accepted an invitation. Invitation-based onboarding
continues to work.

The documented `instant_member_create` entitlement is therefore not considered
complete. A replacement needs one of these product/security designs:

- managed-domain or SCIM provisioning; or
- an admin-created pending record completed by the user through a mailbox-
  verified activation/password flow.

### Direct decision writes

Authenticated clients can no longer mutate leave/access decision state or write
their ledger directly. They must call `decide_leave_request`,
`create_admin_leave_override` or `decide_access_request`. Current UI callers are
migrated in the same change.

### Role permissions

Permission definitions and assignments are owner-managed. Resource assistants
retain only the effective access level configured by the owner; server RPCs now
enforce `approvals` and `admin_override` edit access.

### Payroll export

The UI now delegates calculation, locking and CSV generation exclusively to the
typed `payroll-export` Edge contract. The server validates tenant membership,
admin role, feature entitlement, UUID inputs, provider allowlist and period
state; attendance, leave and rate lookups are deterministically paginated with
explicit safety caps; CSV cells are formula-safe. Open periods use live source
data. Locked/exported periods calculate and export only their stored v1 snapshot,
whose recursively key-sorted canonical JSON and SHA-256 are verified by both Edge
and PostgreSQL. Legacy locked/exported periods without a snapshot fail explicitly
with HTTP 409 / `PAYROLL_SNAPSHOT_MISSING`.

Direct `service_role` period reset and reserved transition-audit mutation are
forbidden. Operational reopen uses `reopen_payroll_period_break_glass`, which is
service-role-only, requires an active workspace payroll admin and an 8–1000
character reason after POSIX-whitespace normalization, locks the authorization
and period rows, archives the exact complete previous protected state and commits
reopen + audit atomically. Runtime `TRUNCATE` is revoked from PUBLIC, anon,
authenticated and service_role on both payroll periods and transition audit.

Direct provider API export intentionally returns HTTP 501 with
`PAYROLL_PROVIDER_API_NOT_IMPLEMENTED`. Lock/export state and primary audit are
now one database transaction. Arbitrary partial-period attendance aggregation,
rate-as-of/effective-to semantics, mixed-currency totals, source reads spanning
multiple HTTP statements and a unique default-provider configuration remain
explicit product and data-model decisions.

The v1 contract currently permits different valid ISO currencies across members
and aggregates their numeric values into one `total_gross`; a contract fixture
explicitly covers EUR+USD acceptance. Financial release therefore requires a
decision between fail-closed single-currency v1 and a versioned per-currency v2
model. Exchange-rate conversion is not implied without a separately audited
rate-source and timestamp contract.

### AuditLog data minimization

The workspace audit UI requests only `id`, `action`, `actor_id`,
`affected_user_id`, `created_at` and `metadata`. It does not transfer protected
prev/new snapshots, IP addresses or user agents to the browser. Rendered
`metadata` remains subject to server-side producer schema and PII minimization.

## Rollback

- Frontend and Edge Functions: redeploy the previous immutable artifact/SHA.
- Database: do not down-migrate by deleting data or policies ad hoc. Restore the
  verified backup for destructive incompatibility, or apply a reviewed forward
  repair that restores the previous RPC/policy contract.
- Keep request/decision/audit/quota ledgers immutable. A rollback must not delete
  or rewrite ledger history.
- Direct user creation remains fail-closed during rollback unless the replacement
  provisioning design is independently approved.
- After applying the payroll snapshot migration, redeploying only the old Edge
  implementation is unsafe. Use the compatible new Edge build or a reviewed
  forward repair; destructive DB rollback requires the verified backup.
- Direct `service_role` reset is not rollback. A single period may be reopened
  only through the audited break-glass RPC with an approved incident reason.

## Known blockers

- Repository/linked migration and schema drift.
- Seven linked DB lint errors.
- Multi-step approval chain configuration is not consumed by the decision RPC.
- Safe replacement for `instant_member_create` is undecided.
- Account deletion/retention is fail-closed but not fully atomic.
- Payroll partial-period, effective-rate, currency and default-provider semantics
  are not approved; source reads are not one MVCC snapshot and legacy snapshotless
  locked/exported periods require inventory and remediation before rollout.
- Direct payroll provider API export is intentionally unavailable (HTTP 501)
  pending a provider and credential contract.
- No production deploy provenance marker exists yet.

payroll semantics, legacy inventory and complete staging evidence.

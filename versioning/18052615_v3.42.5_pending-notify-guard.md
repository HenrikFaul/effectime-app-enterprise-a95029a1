## 1. Pending-notification guard

### Problem
A manager could post a new open shift for the same day+position twice in a row, sending duplicate notifications to the same candidates who were still waiting to respond from the first broadcast.

### Solution

**`useShiftCandidates`** (in `src/hooks/useOpenShifts.ts`) now runs an extra lookup after the main candidate ranking:
1. Query `enterprise_open_shift_requests` for rows matching workspace + shift_date + business_role + status=open + respond_by_at > now()
2. Collect all `notified_user_ids` from those rows
3. Subtract users who have already successfully claimed (`enterprise_open_shift_claims.status = 'claimed'`)
4. Tag each ranked candidate with `pendingNotified: boolean`

Returns `ShiftCandidateResult[]` (= `EligibilityResult & { pendingNotified: boolean }`).

**`OpenShiftManager`** splits the candidate list into:
- `availableCandidates` — selectable, with checkbox, green/amber icon
- `pendingCandidates` — locked, greyed, dashed border, "Folyamatban" clock badge, `title` tooltip explains why

Pending candidates cannot be checked. The "Kiválasztottak értesítése" button only counts available candidates.

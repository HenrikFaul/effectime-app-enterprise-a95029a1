# v3.7.7 — Fix site assignment in time attendance

**Date:** 2026-05-11  
**Branch:** `claude/create-software-documentation-O7kj1`

## Problem

Users reported that setting a site/office in the time-attendance employee day editor silently failed. Two independent root causes:

### Root cause 1 — RLS policies blocked all employee writes

`enterprise_shift_assignments` has INSERT/UPDATE/DELETE policies restricted to `owner` and `resourceAssistant` roles:

```sql
CREATE POLICY "Admins insert shift assignments"
  ON public.enterprise_shift_assignments FOR INSERT TO authenticated
  WITH CHECK (has_enterprise_role(workspace_id, auth.uid(),
    ARRAY['owner'::enterprise_role,'resourceAssistant'::enterprise_role]));
```

Regular `member`-role users (i.e., employees) received a permission error at the database level for any write operation.

### Root cause 2 — CHECK constraint violated on every insert

The table enforces:

```sql
CONSTRAINT shift_role_or_skill CHECK (business_role IS NOT NULL OR skill_id IS NOT NULL)
```

The previous `upsertSiteAssignment` in `api.ts` sent this payload:

```typescript
{ workspace_id, membership_id, user_id, office_id, shift_date, created_by }
```

Neither `business_role` nor `skill_id` was included, so PostgreSQL rejected every insert with a constraint violation before RLS was even evaluated.

## Fix

Created two SECURITY DEFINER RPCs that bypass the restrictive RLS policies while keeping strong ownership validation:

### `attendance_upsert_site_assignment`

- Verifies the calling user (`auth.uid()`) is the owner of `p_membership_id` within the workspace
- Reads their `business_role` from `enterprise_memberships` (satisfies the CHECK constraint; falls back to `'employee'` when the profile has no role assigned)
- Does an atomic `INSERT … ON CONFLICT (workspace_id, user_id, shift_date) DO UPDATE`

### `attendance_remove_site_assignment`

- Allows removal by the membership owner OR a workspace admin (owner/resourceAssistant)
- Deletes by `(workspace_id, membership_id, shift_date)`

### `api.ts` update

`upsertSiteAssignment` and `removeSiteAssignment` now call the RPCs. The `userId` parameter of `upsertSiteAssignment` is retained for signature compatibility but unused (the server resolves identity via `auth.uid()`).

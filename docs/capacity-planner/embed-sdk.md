## Overview

The Effectime **Embed SDK** allows third-party applications — CRM systems, ERP platforms, HR portals, intranet sites — to display live Effectime workforce views inside an `<iframe>` without requiring any user authentication or login. The embedding party obtains a signed token from the Effectime Developer Portal and includes it as a URL parameter in the iframe `src`. Effectime's backend validates the token on every request and returns data scoped to the issuing workspace.

Key capabilities introduced across v3.44.0–v3.46.0:

| Version | Change |
|---------|--------|
| v3.44.0 | Core Embed SDK: `enterprise_embed_tokens` table, `get_embed_view_data` RPC, `EmbedPage`, `EmbedManager` UI, Snippet Builder |
| v3.45.0 | Table name corrections in embed migrations; `EmbedShiftRosterView` added |
| v3.46.0 | Write-enabled tokens (`can_write` column); `embed_assign_shift` / `embed_remove_shift` RPCs; write panel in `EmbedCapacityView`; amber "Write" badge in token list |

## Security Model

The Embed SDK is built on three cooperating layers:

### 1. Signed Token (256-bit hex)

Each embed token is a cryptographically random 256-bit (64-character) hex string generated by PostgreSQL's `extensions.gen_random_bytes(32)` and hex-encoded:

```sql
encode(extensions.gen_random_bytes(32), 'hex')
```

This string is:
- **Unguessable** — 2²⁵⁶ possible values
- **Shown only once** at creation time; never stored in plaintext after initial reveal
- **Stored as a unique index** in `enterprise_embed_tokens.token`

The token ID (UUID) in the iframe URL is **not secret** — it is a public reference. The token string itself is the secret credential.

### 2. SECURITY DEFINER RPCs

All data-access and mutation functions called by embedded views are PostgreSQL `SECURITY DEFINER` functions. They run with elevated privileges and enforce their own authorization logic regardless of the calling role. The calling role is always `anon` (unauthenticated Supabase connection) for embedded views.

Every RPC validates:
- Token exists in `enterprise_embed_tokens`
- `is_active = true`
- `expires_at IS NULL OR expires_at > now()`
- Requested view is in `allowed_views[]`
- For mutations: `can_write = true`

### 3. Anon Supabase Access

Embedded iframes make all API calls as the `anon` Supabase role. No Supabase JWT is issued or required. Row-Level Security (RLS) on underlying tables is bypassed only through the SECURITY DEFINER RPC layer — the `anon` role has no direct table access.

## Quick Start (5 Steps)

### Step 1 — Open Developer Portal

In the Effectime workspace dashboard, navigate to:

**Admin → Developer Portal → Embed tab**

The Embed tab is rendered by `EmbedManager.tsx` and is only visible to workspace owners/admins.

### Step 2 — Create a New Token

Click **"New Token"**. In the Create Token dialog:

- Enter a human-readable **Label** (e.g., `"Salesforce - HQ capacity widget"`)
- Select **Allowed Views**: check `capacity_planner`, `shift_roster`, or both
- Optionally check **"Write-enabled"** to allow the embed to assign and remove shifts
- Optionally set an **Expiry date/time** (leave blank for a perpetual token)

Click **Create**.

### Step 3 — Copy the Token (Shown Once)

After creation, the **New Token Reveal Dialog** appears. It shows the full 256-bit hex token string. **Copy it immediately** — it will never be shown again. Store it securely (password manager, secrets vault, environment variable).

### Step 4 — Open Snippet Builder

Back in the token list, click the **⚙ (gear/snippet) button** on the new token row. The Snippet Builder Dialog opens with:

- **Left panel:** configuration controls (view, office filter, mode, height)
- **Right panel:** live-updating `<iframe>` HTML snippet

Adjust parameters to your desired configuration.

### Step 5 — Paste into Your CRM or Portal

Copy the `<iframe>` HTML from the right panel and paste it into your CRM's HTML widget editor, portal page, or intranet embed block.

## Token Management (EmbedManager UI)

The EmbedManager renders a table of all embed tokens for the workspace. Each row shows:

| Column | Description |
|--------|-------------|
| Label | Human-readable name set at creation |
| Views | Comma-separated list from `allowed_views[]` |
| Mode badge | Amber **"Write"** badge if `can_write = true`; grey **"Read-only"** badge if `can_write = false` |
| Status | Green dot (active) or grey (inactive/revoked) |
| Last used | Timestamp from `last_used_at`; `—` if never used |
| Expires | Expiry datetime or `"Never"` |
| Actions | ⚙ Snippet Builder · ❌ Revoke |

### Token Fields (enterprise_embed_tokens)

| Field | Type | Description |
|-------|------|-------------|
| `id` | uuid | Token identifier (used in URLs, not secret) |
| `workspace_id` | uuid | Owning workspace |
| `created_by` | uuid | Auth user who created the token |
| `label` | text | Human-readable description |
| `token` | text | 256-bit hex secret (unique, stored hashed internally) |
| `allowed_views` | text[] | e.g. `{capacity_planner,shift_roster}` |
| `can_write` | boolean | Whether this token permits shift mutations (v3.46.0) |
| `is_active` | boolean | `false` = revoked; checked on every RPC call |
| `last_used_at` | timestamptz | Updated by `get_embed_view_data` on each call |
| `expires_at` | timestamptz | NULL = perpetual; set to enforce expiry |
| `created_at` | timestamptz | Creation timestamp |

### Revoking a Token

Click **❌** on the token row. A confirmation `AlertDialog` appears. On confirm:

- `revoke_embed_token(_token_id)` RPC is called
- Sets `is_active = false` **instantly** in the database
- Any iframe currently loaded with this token will show **"Invalid token"** on its next data refresh (within the poll interval, typically 30–60 seconds)
- The revocation is permanent; there is no un-revoke function. Create a new token if access is needed again.

### Perpetual vs. Expiring Tokens

- **No expiry** (`expires_at IS NULL`): token is valid indefinitely until revoked
- **With expiry**: token automatically becomes invalid after `expires_at`; iframes show "Token expired" error
- Best practice: use expiring tokens for temporary integrations (e.g., a contractor portal active for 3 months); perpetual tokens for permanent CRM widgets

## URL Format & Parameters

Embedded views are served at the public route:

```
<origin>/#/embed/<view>?token=<token-id>&<additional-params>
```

Full example:

```
https://your-effectime.com/#/embed/capacity_planner?token=a1b2c3d4-e5f6-...&office=550e8400-e29b-...&from=2026-05-19&mode=monthly
```

### Parameter Reference

| Parameter | Required | Type | Description |
|-----------|----------|------|-------------|
| `token` | **YES** | string (uuid) | The token **ID** (UUID), not the raw hex secret. The EmbedPage fetches data using the raw token stored in your CRM config. |
| `office` | NO | uuid | Filter all views to a single office. If omitted, all offices in the workspace are shown. |
| `from` | NO | string (YYYY-MM-DD) | Initial week/month start date. Defaults to current week/month start if omitted. |
| `mode` | NO | `weekly` \| `monthly` | Display mode. Applies to `capacity_planner` only. Defaults to `weekly`. `shift_roster` is always weekly. |

> **Important:** The `token` parameter in the URL is the token **UUID** (e.g., `a1b2c3d4-e5f6-7890-abcd-ef1234567890`), not the 256-bit hex secret string. The hex secret is transmitted as an HTTP header or POST body by the frontend when calling the RPC — it is never embedded in the URL where it could be logged by proxies.

## Views Reference

### capacity_planner

The Capacity Planner embed renders a staffing coverage matrix. It supports two display modes:

#### Weekly Mode (`mode=weekly`)

A standard grid table:
- **Rows:** One row per active coverage rule, grouped by office. Office name appears as a subheader row.
- **Columns:** Days of the selected week (Monday–Sunday by default, respecting workspace locale).
- **Cells:** Each cell shows:
  - **"need / have"** count (e.g., `2 / 3` means 2 needed, 3 assigned)
  - Color coding:
    - Rose/red: understaffed (assigned < needed)
    - Emerald/green: fully staffed or overstaffed
    - Amber/yellow: partially staffed (e.g., 1 of 2 filled)
  - In write mode: clicking a cell opens the write panel below the table

#### Monthly Mode (`mode=monthly`)

A calendar heatmap:
- **Layout:** Full month calendar grid (weeks as rows, days as columns)
- **Cell color:** Represents the **worst-case coverage ratio** across all rules for that date
  - Worst-case = the rule with the lowest `(assigned / needed)` ratio
  - Full green: all rules at 100%+ coverage on that day
  - Amber: at least one rule below 100% but above 0%
  - Rose: at least one rule at 0% coverage (no one assigned)
  - Grey: no rules apply on that day (e.g., weekend with no rules)
- **Navigation:** Previous/Next month buttons in the embed header
- **Write mode:** Click a day to expand into weekly view for that week, then select a cell

### shift_roster

The Shift Roster embed renders a personnel schedule table:

#### Weekly Table Structure
- **Rows:** Employees, grouped by office. Office name appears as a subheader row.
- **Columns:** Days of the selected week
- **Cells:** Each cell shows either:
  - A **role badge** (4-character abbreviation of the business role, e.g., `MNGR`, `CASH`, `STCK`) with the role's color
  - A **dash** (`—`) if no shift is assigned for that member on that day

#### Filtering by Office
- If the `office` URL parameter is set, only employees who have **at least one shift in that office** during the displayed week are shown
- Office grouping is preserved: members without any shift in the filtered office are hidden

#### Write Mode Interactions (v3.46.0)
- **× button** on a shift badge: calls `embed_remove_shift` to delete the assignment
- **+ button** on an empty cell: opens a role/skill selector inline; on confirm calls `embed_assign_shift`

## Snippet Builder

The Snippet Builder is a two-panel dialog accessible via the **⚙ button** on any token row in EmbedManager.

### Left Panel — Configuration Controls

| Control | Description |
|---------|-------------|
| View selector | Dropdown: `capacity_planner` \| `shift_roster`. Updates the URL path. |
| Office filter | Optional: dropdown of workspace offices. Adds `&office=<uuid>` to URL. |
| Mode | `weekly` \| `monthly` (capacity_planner only; hidden for shift_roster) |
| Height | Numeric input, 200–900 pixels. Controls the `height` attribute on the `<iframe>`. |

### Right Panel — Live Snippet Output

Displays a live-updated `<iframe>` HTML block:

```html
<iframe
  src="https://your-effectime.com/#/embed/capacity_planner?token=<token-id>&mode=weekly"
  width="100%"
  height="600"
  style="border:none;border-radius:8px;"
  allowfullscreen
></iframe>
```

The snippet updates in real time as configuration controls are changed. A **Copy** button copies the full snippet to clipboard.

### URL vs. Token Clarification

- The `src` URL contains the **token UUID** (e.g., `token=a1b2c3d4-...`) — this is the token's database ID
- The token UUID is **not secret** — it is a public reference key
- The actual 256-bit hex secret (shown only at creation) is what the frontend stores and sends in RPC calls
- A CRM developer embedding the iframe only needs the `src` URL — they do not need to know the hex secret

## Write-Enabled Tokens

Write-enabled tokens allow the embedded iframe to mutate shift assignments directly — assigning members to shifts and removing existing assignments — without requiring the user to log into Effectime.

### Creating a Write-Enabled Token

1. In the Create Token dialog, check **"Write-enabled"** (`can_write = true`)
2. All other fields (label, views, expiry) work identically to read-only tokens
3. After creation, the token list shows an amber **"Write"** badge in the Mode column

### Visual Indicators

| Location | Indicator |
|----------|-----------|
| EmbedManager token list | Amber "Write" badge vs. grey "Read-only" badge |
| Embedded capacity planner header | Amber "✏ szerkesztés" (edit) badge visible to end users |
| Embedded shift roster | + / × buttons visible on cells |

### Capacity Planner Write Flow

1. Click any coverage cell in the embedded capacity planner
2. A **write panel** appears below the grid (inline, not a modal)
3. The write panel has two columns:
   - **BEOSZTOTT** (Assigned): list of currently assigned members with **×** remove buttons
   - **ELÉRHETŐ** (Available): list of eligible available members with **+** assign buttons
4. Clicking **×** calls `embed_remove_shift(_token, _assignment_id)` RPC
5. Clicking **+** calls `embed_assign_shift(_token, _user_id, _office_id, _business_role, _shift_date)` RPC
6. Both update the grid view in real time

### Shift Roster Write Flow

- **× button** on a shift badge: immediately removes that assignment
- **+ button** on an empty cell: opens an inline mini-form to select role/skill; confirms with `embed_assign_shift`

### Server-Side Write Authorization

All write RPCs enforce `can_write = true` server-side even if a client somehow calls them with a read-only token:

```sql
IF NOT (SELECT can_write FROM enterprise_embed_tokens WHERE token = _token AND is_active) THEN
  RAISE EXCEPTION 'Token does not have write permission';
END IF;
```

This means: a read-only token **cannot** be used to write even by a technically sophisticated attacker who reverse-engineers the RPC signature.

## Security Considerations

### Token Secrecy

| Asset | Secret? | Notes |
|-------|---------|-------|
| Token UUID (in URL) | No | Public reference; safe to log, cache, share in HTML source |
| 256-bit hex token string | **YES** | Treat like an API key; store in secrets manager |

The hex string is shown **once** at creation. If lost, revoke and create a new token — there is no "show token again" function.

### Storage Recommendations

- Store the hex token in your CRM's encrypted secrets store or environment variables
- Do not commit the hex token to version control
- Do not log it in application logs
- The token UUID (used in iframe `src`) can be stored in plain config — it is not sensitive

### Token Rotation

When rotating a token (periodic security hygiene or suspected compromise):

1. Create a new token with identical settings
2. Update the iframe `src` in your CRM/portal config (change the UUID)
3. Verify the new embed loads correctly
4. Revoke the old token via EmbedManager ❌

There is no zero-downtime atomic rotation — there will be a brief period where either the old or new token is active. For minimizing disruption: create the new token first, update the CRM config, verify, then revoke the old token.

### Compromised Token Response

If a token is suspected compromised (e.g., found in public HTML source, leaked log):

1. **Immediately** click ❌ on the token in EmbedManager
2. Confirm revocation — `is_active = false` is applied instantly
3. Any iframe using the compromised token will fail on next data poll
4. Create a replacement token and update your CRM configuration

### Write Token Additional Considerations

- Write-enabled tokens are higher-risk than read-only tokens
- Scope write tokens to the minimum required `allowed_views`
- Consider using the `office` URL parameter to limit what the write token can affect (though this is URL-enforced, not server-enforced — the server enforces workspace scoping)
- Audit `last_used_at` regularly to detect unexpected usage

### No Direct Table Access

The `anon` Supabase role used by embedded iframes has **no direct table SELECT/INSERT/UPDATE/DELETE permissions** on any Effectime table. All access goes through SECURITY DEFINER RPCs which enforce workspace scoping on every call.

## Database RPCs Reference

All RPCs are PostgreSQL functions accessible via the Supabase PostgREST API. Embedded views call them as the `anon` role.

---

### create_embed_token

Creates a new embed token. Returns the token ID and the raw hex string (shown once).

```sql
FUNCTION public.create_embed_token(
  _workspace_id  uuid,
  _label         text,
  _allowed_views text[]      DEFAULT '{capacity_planner}',
  _expires_at    timestamptz DEFAULT NULL,
  _can_write     boolean     DEFAULT false
) RETURNS TABLE (
  id    uuid,
  token text
)
LANGUAGE plpgsql
SECURITY DEFINER
```

**Caller:** Authenticated workspace owner/admin only. The function validates that the calling `auth.uid()` has the `owner` or `admin` role for `_workspace_id`.

**Returns:** A single row with:
- `id`: The token UUID (safe to embed in URLs and config)
- `token`: The 256-bit hex secret string — **store this immediately; it will not be shown again**

**Example (Supabase JS client):**

```typescript
const { data, error } = await supabase.rpc('create_embed_token', {
  _workspace_id: workspaceId,
  _label: 'Salesforce CRM widget',
  _allowed_views: ['capacity_planner'],
  _expires_at: null,
  _can_write: false,
});
// data[0].id   = token UUID (put in iframe src)
// data[0].token = hex secret (store securely, show to admin once)
```

### get_embed_view_data

Fetches all data needed to render an embedded view. Called by `EmbedPage` on load and on navigation.

```sql
FUNCTION public.get_embed_view_data(
  _token     text,   -- raw 256-bit hex token string
  _view      text,   -- 'capacity_planner' | 'shift_roster'
  _from_date date,
  _to_date   date
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
```

**Caller:** `anon` role (no authentication required).

**Validates:**
- Token exists in `enterprise_embed_tokens`
- `is_active = true`
- `expires_at IS NULL OR expires_at > now()`
- `_view = ANY(allowed_views)`

**Side effect:** Updates `last_used_at = now()` on the token row.

**Returns:** A `jsonb` object with the following structure:

```jsonc
{
  "workspace_id": "<uuid>",
  "can_write": false,
  "offices": [
    { "id": "<uuid>", "name": "Budapest HQ", "timezone": "Europe/Budapest" }
  ],
  "coverage_rules": [
    {
      "id": "<uuid>",
      "office_id": "<uuid>",
      "name": "Morning Shift",
      "business_roles": ["cashier"],
      "skill_ids": [],
      "min_headcount": 2,
      "days_of_week": [1,2,3,4,5]
    }
  ],
  "shift_assignments": [
    {
      "id": "<uuid>",
      "user_id": "<uuid>",
      "office_id": "<uuid>",
      "shift_date": "2026-05-19",
      "business_role": "cashier"
    }
  ],
  "members": [
    {
      "membership_id": "<uuid>",
      "user_id": "<uuid>",
      "display_name": "Kovács Anna",
      "business_role": "cashier",
      "weekly_capacity_hours": 40
    }
  ],
  "holidays": ["2026-05-21"],
  "blocked_dates": ["2026-05-25"],
  "leave_requests": [
    {
      "user_id": "<uuid>",
      "start_date": "2026-05-19",
      "end_date": "2026-05-20",
      "status": "approved"
    }
  ]
}
```

### embed_assign_shift

Creates a new shift assignment through an embed token. Requires `can_write = true`.

```sql
FUNCTION public.embed_assign_shift(
  _token         text,
  _user_id       uuid,
  _office_id     uuid,
  _business_role text,
  _shift_date    date,
  _skill_id      uuid DEFAULT NULL
) RETURNS uuid
LANGUAGE plpgsql
SECURITY DEFINER
```

**Caller:** `anon` role (no authentication required; token validates write permission).

**Validates:**
- Token is active and not expired
- `can_write = true`
- `_office_id` belongs to the token's workspace (prevents cross-workspace writes)
- `_user_id` is an active member of the workspace

**Returns:** The UUID of the newly created `enterprise_shift_assignments` row.

**Errors:**
- `Token is inactive or expired` — if token is revoked or past `expires_at`
- `Token does not have write permission` — if `can_write = false`
- `Office not in workspace` — if `_office_id` doesn't belong to the workspace
- `User not a workspace member` — if `_user_id` is not in `enterprise_memberships`

---

### embed_remove_shift

Deletes an existing shift assignment through an embed token. Requires `can_write = true`.

```sql
FUNCTION public.embed_remove_shift(
  _token         text,
  _assignment_id uuid
) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
```

**Caller:** `anon` role.

**Validates:**
- Token is active and not expired
- `can_write = true`
- Assignment belongs to the token's workspace (prevents cross-workspace deletion)

**Effect:** Hard-deletes the `enterprise_shift_assignments` row identified by `_assignment_id`.

---

### revoke_embed_token

Revokes a token, making it permanently invalid.

```sql
FUNCTION public.revoke_embed_token(
  _token_id uuid
) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
```

**Caller:** Authenticated workspace owner/admin only.

**Effect:** Sets `is_active = false` on the token row. The change takes effect immediately — the next call to `get_embed_view_data` or any write RPC with this token will return an error.

**Note:** Revocation is irreversible. There is no `restore_embed_token` function. Create a new token if access needs to be reinstated.

## Integration Guide for CRM Developers

This section is for developers integrating Effectime embeds into external systems (Salesforce, HubSpot, SAP, custom portals, etc.).

### Step-by-Step Integration

**1. Obtain the iframe snippet**

Ask your Effectime workspace admin to:
- Create an embed token in Developer Portal → Embed
- Open the Snippet Builder for that token
- Configure the view, office filter, mode, and height
- Copy the `<iframe>` HTML snippet

You do not need access to the Effectime application to embed the view — only the snippet.

**2. Paste into your CRM's HTML widget**

Most CRM/portal platforms support a generic "HTML widget", "embed block", or "web component" area. Paste the `<iframe>` HTML directly. Common locations:

| Platform | Where to paste |
|----------|----------------|
| Salesforce | Lightning App Builder → HTML component |
| HubSpot | Website Pages → Custom HTML module |
| SharePoint | Text web part → "Edit source" / Embed web part |
| Notion | `/embed` block |
| WordPress | Custom HTML block |
| Generic intranet | Any HTML editor accepting `<iframe>` |

**3. Test with the VibeCoding Embed Tester**

The VibeCoding Embed Tester tool (available in Effectime's developer resources) allows you to:
- Paste an iframe snippet and see a live preview
- Break down the URL parameters for debugging
- Verify token validity (shows "Token valid ✓" or error details)
- Test write operations if using a write-enabled token

**4. For write-enabled tokens: train your users**

If the embed uses a write-enabled token, users interacting with the iframe will be modifying live Effectime data. Ensure they understand:
- Shift assignments made via the embed are real and immediately visible to all Effectime users
- There is no draft/preview mode — changes are applied immediately
- Removals via × button are permanent (no undo in the embed UI)

**5. Token rotation strategy**

We recommend:
- Store the token **label** and **UUID** in your CRM configuration (both are safe to store)
- Store the hex secret in an encrypted secret store (not in CRM config fields)
- Schedule annual token rotation as a recurring task
- When the Effectime admin rotates the token, they provide you with a new iframe snippet (new UUID in the `src`); update your CRM widget accordingly

### Responsive Sizing

The embed is designed to fill its container width. Set `width="100%"` and choose a `height` that fits your layout:

| Use case | Recommended height |
|----------|--------------------|
| Capacity planner (weekly, 3–5 rules) | 400–600px |
| Capacity planner (monthly heatmap) | 500–700px |
| Shift roster (5–15 employees) | 500–800px |
| Dashboard widget (compact) | 300px (weekly mode, office-filtered) |

### Handling Token Expiry in Long-Lived CRM Widgets

If a token expires while an iframe is displayed:
- The embedded view will show an error message on the next data refresh
- There is no automatic token refresh mechanism
- The CRM admin must obtain a new iframe snippet from the Effectime admin and update the widget

To prevent unexpected expiry, prefer **perpetual tokens** for permanent CRM integrations and use expiring tokens only for time-limited use cases.

### Content Security Policy (CSP)

If your CRM or portal enforces a Content Security Policy, you may need to allowlist the Effectime origin:

```
frame-src https://your-effectime.com;
```

Adjust `your-effectime.com` to your actual Effectime deployment origin.

### Cross-Origin Considerations

The embed iframe is served from the same Effectime origin as the main application. No special CORS configuration is needed for the iframe itself — CORS only applies to `fetch`/`XMLHttpRequest` calls, not `<iframe>` src loads. The RPCs called by the embedded views are subject to Supabase's CORS configuration, which is managed by the Effectime platform team.
